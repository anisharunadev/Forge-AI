## 3. Anti-pattern surface

Spot-check of the qualitative §3 acceptance-bar criteria from the README:

- **"No tribal knowledge."** — yes; every concept (approval gate, audit trail, `mode: live|sample`) is either defined in-line or pointed at a memory/customer file via the §"Related" section.
- **"Opinionated. 'It depends' is not in a v1 file."** — yes; the file is unambiguously prescriptive ("Synthesizer is rule-based, not LLM-backed", "The approval gate is a callable, not a class hierarchy").
- **"`Related` section at the bottom."** — yes; lines 64-70.

## 4. Structural finding — section numbering

`ideation.md` is the **only** memory file in the spine without numbered top-level sections (`## What this stage does`, `## How to invoke`, ...). Every other memory file ([architecture.md](../architecture.md), [coding.md](../coding.md), [devops.md](../devops.md), [qa.md](../qa.md), [security.md](../security.md)) and every customer/project file uses `## N. Title` numbering. This is the source of a second, real cross-reference defect:

- [`qa.md` §4 #1](../qa.md#4-non-obvious-design-choices) reads: *"Same reasoning as the Ideation synthesizer (see `ideation.md` §4)"* — but `ideation.md` has **no §4**; the matching content lives in the unnumbered "Non-obvious design choices" section. A cold-started QA sub-agent following that pointer lands at the wrong place.

**Decision in this audit:** the fix is documented and filed as a follow-up child issue rather than executed inline, because adding section numbers to `ideation.md` is a real edit (not a typo fix) and risks a section-number drift across files that already cite it. The footer of `ideation.md` is patched in this same pass to call this out so a cold-started reader is not surprised.

Filed as: child of FORA-242 — *"Number ideation.md sections to match the spine and update qa.md §4 #1 pointer"*.
