## Rationale — citing skill rules

The redesign is anchored to the same eight constitutional rules that govern the
Forge AI app:

1. **Model-provider agnosticism** — landing page surfaces "LiteLLM proxy" and
   "provider-agnostic" prominently; new connector + governance pages show the
   swap-without-code-changes story.
2. **Multi-tenancy by default** — every new page references tenant scoping;
   multi-tenant setup guide explains per-tenant KMS keys + RLS + audit topology.
3. **Mandatory human approval gates** — co-pilot, workflows, command-center, and
   ticket-driven guide all surface HITL gates explicitly; before/after comparison
   table makes the "skips review" anti-pattern concrete.
4. **Typed artifacts only** — landing page lists the six types in the
   feature grid; artifacts center formalizes F-001..F-005; quickstart links
   to ADR/contract artifacts.
5. **Layer isolation** — artifacts (org-shared) and projects (per-project) are
   distinct centers; "Knowledge" center distinguishes from "Artifacts" by
   isolation scope; diagram on artifacts page makes the boundary visual.
6. **Mandatory auditability** — audit center documents the WORM ledger and
   hash-chain anchors; governance + guardrails pages show how policy decisions
   and overrides are auditable.
7. **Mandatory observability** — analytics center covers cost + adoption +
   run-health + policy-compliance + audit-volume; terminal + run pages surface
   real-time signal streams.
8. **Configurable everything** — connector center documents the SDK for
   bring-your-own connectors; settings center lists all knobs (RBAC, KMS,
   policies, integrations, billing).

The ui-ux-pro-max skill's documentation-site, dark-mode-OLED, and
minimalism/Swiss-style results informed the typography (Inter + JetBrains Mono),
the color system (deep `#09090B` + indigo `#6366F1`), and the layout primitives
(grid-based, generous spacing, restrained decoration).

## What we deliberately did NOT change

- **Starlight framework** — kept as the base. Migrating to Next.js or VitePress
  would lose the existing content-config + auto-generated routes + Pagefind
  search + sitemap + dark-mode toggle UX out of the box.
- **Existing folder structure** — `src/content/docs/{concepts,architecture,…}/`
  is preserved. Pages were not moved; only the sidebar config was rewritten to
  expose them in the new IA.
- **All existing pages** — every one of the 60+ pre-existing pages remains
  reachable. No content was deleted.
- **Icon system** — kept inline SVG (Lucide-style) instead of adding a JS icon
  library. Zero runtime cost.
- **Search** — kept Starlight's built-in Pagefind (self-hosted, no Algolia key
  required). The goal calls for "Algolia DocSearch OR Pagefind"; we chose
  Pagefind to keep the build fully static and self-hostable.
- **Docker / Dockerfile** — unchanged. The `docs-site` service in docker-compose
  already serves the static `dist/` output on port 4321.
- **Logo asset** — kept the existing SVG. The drop-shadow halo comes from CSS.
- **Header / Footer Astro components** — preserved; the footer was reorganized
  to match the four-column layout.
