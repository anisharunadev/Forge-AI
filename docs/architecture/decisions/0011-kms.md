# ADR-011: Pilot-vs-Multi-Tenant KMS Topology

- Status: Accepted
- Date: 2026-06-26
- Deciders: Forge Architecture Working Group
- Related research: docs/research-forge-architecture-decisions-2026-06-20.md (Q4 multi-tenant rollout topology, Q6 KMS layering)

## Context and Problem Statement

Forge AI's enforcement boundary for the M2 substrate lock — between **pilot mode** (single-tenant managed-service deployments) and **multi-tenant (MT) mode** (many tenants sharing the same control plane under RLS) — extends to the cryptographic layer. Customer data at rest is encrypted with AWS KMS Customer Master Keys (CMKs); the question is **how many CMKs, and how they are scoped**.

Three operating regimes are possible:

1. **Single CMK for the whole deployment.** Cheap, simple, devastating blast radius — any compromise unlocks every customer's data plane.
2. **Per-tenant CMK from day 1.** Cryptographic isolation matches the MT isolation posture (Rule 2, NFR-006), but it forces every pilot tenant to provision its own KMS key grant before the first commit lands. Pilot customers regularly stall at this step and never onboard.
3. **Hybrid.** Pilot tenants share a single CMK while tenant_count == 1; the deployment graduates to per-tenant CMKs once the second tenant crosses the pilot cutoff defined in ADR-010 (`tenant_count >= 2`). The cutoff is irreversible.

The forces at play:

- Pilot customers (tenant_count == 1) expect a managed-service onboarding experience: a single KMS grant, a single signing key, no requirement to also produce a customer-managed KMS alias.
- MT customers (tenant_count >= 2) expect strict cryptographic isolation: a stolen `aws_secret_access_key` from one tenant must not decrypt any other tenant's storage.
- The transition from pilot to MT is irreversible (ADR-010 Appendix B), so the KMS topology decision is also irreversible in any surviving deployment.
- KMS grants are AWS-account-scoped; per-tenant CMKs require the KMS key policy to grant `kms:Decrypt` to the Forge service role + the tenant's IAM principal.
- Pilot tenants may be invited design partners; their negotiated contracts (DSAs) cover the cryptographic boundary explicitly.

We must choose a single KMS topology that holds across both operating modes.

## Decision Drivers

- Rule 2: Multi-tenancy — every tenant-scoped table carries `tenant_id` + `project_id`; cryptographic isolation mirrors that boundary where it does not block onboarding.
- ADR-001: Cloud-only AWS deployment (KMS = AWS KMS; no Bring-Your-Own-Key requirement for the pilot rollouts).
- ADR-002: PostgreSQL 17 + Apache AGE + pgvector — Aurora storage encryption uses the chosen CMK for the entire cluster; per-tenant CMKs apply to tenant-scoped object storage (S3) and selective column-level encryption, not to the Aurora cluster itself.
- ADR-010: Pilot-vs-MT conflict-resolution policy — defines the `tenant_count >= 2` cutoff that flips the topology from shared CMK to per-tenant CMKs.
- NFR-006: Tenant isolation, including cryptographic isolation in MT mode.
- NFR-002: GDPR — key rotation must be tenant-scoped in MT mode so a deletion request maps cleanly to a key rotation.
- NFR-001: SOC2-controls-ready posture; the chosen topology must be auditable in a single KMS key policy.

## Considered Options

- **A. Single CMK for everything** — one `forge-shared` CMK encrypts pilot data, MT data, and shared services. Cheapest onboarding; worst blast radius.
- **B. Single CMK for pilot + per-tenant CMK from tenant #3** *(chosen)*
- **C. Per-tenant CMK from day 1** — every tenant gets its own CMK even before tenant_count crosses the cutoff. Highest isolation; blocks pilot onboarding.

## Decision Outcome

Chosen option: **B — single CMK for the pilot regime (tenant_count < `PER_TENANT_CMK_THRESHOLD`); per-tenant CMK from tenant `#(per_tenant_cmk_threshold)` onward (default threshold = 3)**.

Concretely:

- A new setting `settings.per_tenant_cmk_threshold: int = 3` controls the cutoff. The default matches the desired operator experience: tenant #1 and #2 share `forge-shared-pilot`; tenant #3 and beyond each receive their own KMS CMK on enrollment.
- Pilot tenants (count < threshold) get a single shared CMK named `forge-shared-pilot-<region>` with a key policy that grants `kms:Decrypt` to the Forge service role and to the tenant's IAM principal. The key is rotated annually via AWS-managed rotation.
- MT tenants (count >= threshold) each receive their own `forge-tenant-<tenant_id_hash>` CMK on enrollment. The key policy grants `kms:Decrypt` only to (a) the Forge service role, scoped to the tenant's `tenant_id` tag, and (b) the tenant's IAM principal. The key is rotated annually AND on any tenant-side key rotation request (NFR-002).
- Aurora cluster encryption uses the AWS-managed default key; per-tenant CMKs apply to S3 (tenant artifact buckets), per-tenant audit exports, and DynamoDB tables (workspace state). Aurora row-level isolation remains the responsibility of RLS (ADR-002) + tenant_id RLS, not the KMS layer.
- The threshold evaluation is checked at tenant enrollment (via `backend/app/services/tenant_lifecycle.py` — owned by Tracks A/B) and again at every `forge-*` command that touches encrypted storage (cheap integer compare against `tenants.config.kms_mode`).
- Tenant enrollment at the threshold boundary is atomic: the KMS CMK is provisioned BEFORE the tenant row is committed; on CMK provisioning failure the transaction rolls back and the operator sees a recorded `tenant_enrollment.failed_kms_provisioning` audit row.

### Consequences

Positive:

- Pilot customers onboard without bespoke KMS setup (the friction that historically blocked pilot conversion).
- MT customers get cryptographic isolation matching their contractual promise as soon as they reach the third tenant — the cutoff is deterministic and matches the pilot-MT transition rule in ADR-010 Appendix B.
- Per-tenant CMK rotation is the GDPR primitive — rotation for deletion maps cleanly to a tenant's CMK, satisfying NFR-002 (right-to-erasure) without disturbing other tenants.
- The threshold is configurable (`settings.per_tenant_cmk_threshold`), so a deployment that wants per-tenant CMK from day 1 (rare, but documented) sets the threshold to 1; a deployment that wants MT-like isolation from tenant #10 onward sets the threshold to 10.

Negative:

- Two operating modes inside the same code path means the encryption layer must accept both `kms:alias/forge-shared-pilot-*` and `kms:alias/forge-tenant-*` aliases and resolve to the right CMK at runtime. A bug here is a security-class incident, not a perf incident.
- Tenants #1 and #2 (when the threshold default holds) share the `forge-shared-pilot` CMK — a key compromise unlocks both customers' artifact storage. The annual rotation limits the exposure window but does not eliminate it.
- Per-tenant CMK provisioning is a synchronous step in tenant enrollment; AWS KMS quotas can throttle this at scale. The implementation must back off + retry and surface the throughput-exceeded case to the operator rather than rolling back enrollment silently.

Neutral:

- The threshold default of 3 may be reconsidered when more pilot data is available; changing it requires an ADR (e.g. ADR-014) because changing the threshold retroactively would force CMK rotation for tenants #1 and #2 — a WORM-relevant change.

## Alternatives Considered

### Single CMK for everything

Pros: simplest to operate; one KMS key policy; one annual rotation.

Cons: any compromise decrypts every tenant's data — there is no GDPR-compliant path for per-tenant erasure; vendor KMS access for one tenant unlocks all tenants; the SOC2 audit reviewers will flag the shared-key model immediately.

Rejected: violates NFR-002 (GDPR right-to-erasure) and NFR-006 (tenant isolation). The blast radius is unacceptable for any deployment past the pilot regime.

### Per-tenant CMK from day 1

Pros: every tenant gets cryptographic isolation from the first commit; the SOC2 audit story is identical to the MT story; there is no in-place topology transition to engineer.

Cons: the pilot customer onboarding flow stalls at "please provision a customer-managed KMS key and grant `kms:Decrypt` to our service role" — multiple pilots in the prior gen stalled at this step and never landed. The pilot population is small but high-value; losing pilot conversions for a 0.001% security-class edge case over the first three tenants is a poor trade.

Rejected: optimizes for an edge case the pilot population cannot reach; the ADR-010 cutoff rule already provides a clean migration step at tenant #3.

## Pros and Cons of the Chosen Option

Pros:

- Pilot onboarding stays friction-free (tenant_count < threshold uses the shared CMK).
- MT isolation (per-tenant CMK) is automatic and irreversible once the cutoff fires.
- Threshold is configurable via `settings.per_tenant_cmk_threshold` — operators can tighten or loosen the pilot latitude without an ADR if they accept a small WORM-relevant risk; changing the default value of 3 does require an ADR.
- Per-tenant CMK rotation covers GDPR right-to-erasure (NFR-002) cleanly.

Cons:

- The shared CMK covers up to two pilot tenants by default — that blast-radius window should be acknowledged in every pilot customer's DSA.
- Per-tenant CMK provisioning is a synchronous enrollment step; KMS quotas can throttle the bootstrap path.
- Two operating modes inside one code path = two KMS alias shapes + two key policies to review per security audit.

## Appendix A — Topology diagrams

Pilot regime (tenant_count < `per_tenant_cmk_threshold`, default 2):

```
                        +----------------------------------+
                        |          AWS Account A           |
                        | (Forge control-plane account)    |
                        |                                  |
    +-----------------+ |    +------------------------+    |
    |  Tenant #1 IAM  |-+--->| KMS CMK                |    |
    +-----------------+ |    |  alias:                |    |
                        |    |   forge-shared-pilot-  |    |
    +-----------------+ |    |   us-east-1            |    |
    |  Tenant #2 IAM  |-+--->|  rotation: annual      |    |
    +-----------------+ |    |  grants:               |    |
                        |    |   - Forge svc role     |    |
                        |    |   - Tenant #1 IAM      |    |
                        |    |   - Tenant #2 IAM      |    |
                        |    +------------------------+    |
                        |                                  |
                        |  encrypts:                       |
                        |    S3 (tenant_artifacts/*)       |
                        |    DynamoDB (workspace_state/*)  |
                        |    audit-export bundles          |
                        +----------------------------------+
```

MT regime (tenant_count >= `per_tenant_cmk_threshold`, default 3):

```
                        +--------------------------------------------------+
                        |              AWS Account A                        |
                        |        (Forge control-plane account)              |
                        |                                                  |
                        |  per-tenant CMKs (tenant #3 onward):              |
                        |                                                  |
                        |   +----------------------+   +--------------------+|
                        |   | KMS CMK tenant #3   |   | KMS CMK tenant #4  ||
                        |   |  forge-tenant-<h3>  |   |  forge-tenant-<h4> ||
                        |   |  rotation: annual   |   |  rotation: annual  ||
                        |   |  grants:            |   |  grants:           ||
                        |   |   - Forge svc role  |   |   - Forge svc role ||
                        |   |     tag tenant=#3   |   |     tag tenant=#4  ||
                        |   |   - Tenant #3 IAM   |   |   - Tenant #4 IAM  ||
                        |   +----------------------+   +--------------------+|
                        |                                                  |
                        |   (tenants #1, #2 still share pilot CMK until    |
                        |    their next material artifact write, at which   |
                        |    point the topology re-encrypts under their own |
                        |    CMK and the pilot CMK is retired per key      |
                        |    rotation policy)                              |
                        +--------------------------------------------------+
```

Aurora cluster (always) and shared services (always) use a separate `forge-shared-infra` CMK that is independent of any tenant's data plane. This diagram is in line with ADR-001 (AWS-only) and ADR-002 (RLS as the per-row isolation primitive); per-tenant CMKs are additive, not a replacement for RLS.

## Appendix B — Per-tenant CMK rollout runbook (5 steps)

Threshold default value: `settings.per_tenant_cmk_threshold: int = 3`. The first `threshold - 1` tenants (default: #1 and #2) share `forge-shared-pilot-<region>`; tenant `#threshold` and beyond each receive their own `forge-tenant-<tenant_id_hash>` CMK at enrollment.

The 5-step runbook for onboarding a tenant across the threshold (executed by `backend/app/services/tenant_lifecycle.py`, owned by Track A/B):

1. **Pre-flight (synchronous, pre-DB).** Compute the current tenant count `n` via the same SQL aggregate that ADR-010 Appendix B uses (`SELECT COUNT(DISTINCT tenant_id) FROM audit_log WHERE occurred_at > NOW() - INTERVAL '24 hours'`). Decide `mode = "pilot" if n < settings.per_tenant_cmk_threshold else "mt"`. Reject the enrollment with HTTP 503 if AWS KMS API health is degraded (`DescribeKey` fails or throttles 5xx); this is a synchronous step so the operator must see the failure.

2. **CMK provisioning (synchronous).**
   - If `mode == "pilot"`: validate that `forge-shared-pilot-<region>` exists; if it does not (first-ever pilot), create it with the standard key policy from Appendix A's pilot diagram, enable AWS-managed annual rotation, and apply the `forge:tenant-mode=pilot` tag.
   - If `mode == "mt"`: create a new CMK named `forge-tenant-<sha256(tenant_id)[:12]>` with the per-tenant key policy from Appendix A's MT diagram, enable annual rotation, apply the `forge:tenant-id=<tenant_id>` tag, and record the CMK ARN in the new tenant row's `kms_key_arn` column. Backoff + retry on `ThrottlingException` (5 retries with exponential delay); surface `tenant_enrollment.failed_kms_provisioning` audit row on exhaustion.

3. **Tenant row commit (transactional).** Within a single DB transaction: insert the tenant row with `kms_mode = mode`, `kms_key_arn = (shared_alias or new_cmk_arn)`, and `config.mode` set to either `"pilot"` (always — tenants #1, #2) or `"mt"` (always — tenants >= threshold). The transaction commits only if the KMS provisioning in step 2 succeeded AND the row insert succeeded; either failing rolls both back.

4. **Audit trail (synchronous).** Write an F-005 audit row at action `tenant_enrollment.completed` with payload `{tenant_id, kms_mode, kms_key_arn, per_tenant_cmk_threshold, computed_tenant_count}`. The row enters the daily chain (ADR-008); the SOC2 reviewer can confirm that every tenant has an audit trail entry that names its KMS configuration.

5. **Materialization (deferred, async).** For any tenant newly assigned an MT-mode CMK, schedule a background job (`backend/app/services/scheduler/service.py`) that re-encrypts the tenant's artifact blobs under the new CMK. Until that job completes, the tenant may continue reading through the pilot CMK for backward compatibility; after completion, the `forge-shared-pilot-*` grant for that tenant is removed. The job reports progress via a second F-005 row at action `tenant_kms_rotation.completed`.

The 5 steps complete in < 30 seconds for a healthy AWS region; KMS throttling extends the window but never blocks enrollment beyond the configured retry budget. The per-tenant CMK ARN is the canonical handle other code paths use (`S3 PutObject` on tenant buckets, DynamoDB encrypt-at-rest, audit-export bundles) — there is no central alias-resolution layer, so a typo or stale alias is a debug-time problem rather than a runtime surprise.
