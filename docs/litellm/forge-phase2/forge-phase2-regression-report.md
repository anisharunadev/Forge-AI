## What Phase 2 changes

1. **New pre-call envelope** in `ForgeLLMClient.chat`:
   - `GuardrailsService.apply(kind="pre_call_input", guardrail_names=policies_service.resolve(...).effective_guardrails)` runs BEFORE the chat request.
2. **New post-call envelope** in `ForgeLLMClient.chat` (streaming):
   - Same service, `kind="post_call_output"` on accumulated output; `kind="during_call"` per chunk.
3. **Tool dispatch** is now in the chat loop:
   - When the model emits `tool_calls`, `MCPService.dispatch_tool_call` is called per tool.
4. **Skill injection** happens once per chat:
   - `SkillsService.inject(skill_ids, request)` calls `/utils/transform_request` once per skill.

## What Phase 2 does NOT change

- Spend writes (`spend_logs`).
- Virtual key minting / rotation.
- Audit log writing paths (Phase 2 writes through the same `audit_service.record`).
- CORS / telemetry / event bus transport (only new event types added).
- SSE wire format (same `data: <chunk>\n\n`).

## Smoke runbook

```bash
cd backend && source .venv/bin/activate

# 1. P1 regression — readiness probe
uvicorn app.main:app --port 8000 &
sleep 5
curl -fsS http://localhost:8000/api/v1/forge/health | jq

# 2. P5 regression — SSE chat with guardrails applied
curl -fsS -N http://localhost:8000/api/v1/forge_chat/stream \
  -H "Authorization: Bearer $JWT" \
  -d '{"messages":[{"role":"user","content":"hello"}],"model":"claude-sonnet-4-6"}'

# 3. Phase 2 — guardrail register + apply
curl -fsS -X POST http://localhost:8000/api/v1/guardrails \
  -H "Authorization: Bearer $ADMIN_JWT" \
  -d '{"guardrail_name":"smoke_pii","kind":"pre_call_input","litellm_params":{"pii_entities":["EMAIL_ADDRESS"]}}'

# 4. Phase 2 — policies resolve
curl -fsS -X POST http://localhost:8000/api/v1/policies/resolve \
  -H "Authorization: Bearer $JWT" \
  -d '{"tenant_id":"<tid>"}'

# 5. Phase 2 — skills list
curl -fsS http://localhost:8000/api/v1/skills -H "Authorization: Bearer $JWT"
```
