## 2026-06-19 — Revisions to vision (Coaching path confirmed)

User accepted all 7 pushback points and filled the missing context. Major reframings captured below.

### Naming
- Renamed: `Forge SDLC Platform` → **`Forge Delivery Accelerator`**
- Rationale: leadership buys delivery outcomes, not "an SDLC platform."

### Goals (revised)
1. **Reduce lead time** from approved Jira story to merged PR.
   - Baseline: KnackForge project historical average.
   - Target: 25-30% reduction.
2. **Reduce engineering rework** caused by late requirement/architecture changes.
   - Signals: story reopen rate, sprint spillover %, ADR modifications after approval, production defects from misunderstood requirements.
3. **Increase delivery predictability** across customer engagements.
   - Signals: onboarding time, architecture review effort, project-setup consistency, team-to-team variation.
4. Reusable accelerator (kept, clarified as phased).
5. Standardization → reframed as predictability (means, not goal).

### Product strategy (revised)
- **Not 7 flat modules.** Customer-facing entry points (outcomes):
  - Architecture Accelerator (Architects, Tech Leads)
  - Development Accelerator (Developers)
  - Modernization Accelerator (Solution Architects)
- Upsell path: Architecture → Development → QA → Security.
- Phased rollout:
  - Phase 1: Internal Accelerator
  - Phase 2: Customer-facing Accelerator (bundled into KnackForge services)
  - Phase 3: Commercial Product

### MVP (revised)
- First user: **KnackForge Technical Leads / Architects** (not admin, not external).
- First aha: **Jira Story → Architecture → ADR → Task Breakdown in minutes**.
- Phase 1 ship: Architecture Accelerator (single killer feature).
- Phase 2: Development Accelerator.
- Phase 3: Security + QA.
- Phase 4: Full SDLC Platform.

### Architecture posture
- Brief will describe **capabilities** (workflow orchestration, artifact management, human approvals, agent execution, auditability, memory retrieval, external integrations), not technologies.
- Tech choices (LangGraph, FastAPI, PostgreSQL 17, Keycloak, etc.) move to ADRs.

### Stakes / budget
- Primary: Internal Accelerator for KnackForge.
- Budget source: Innovation / Engineering Excellence (not product revenue).
- Long-term: customer-facing, then commercial.

### Why now
- Every customer asks "how are you using AI?" (not "are you").
- Competitive pressure: Cursor, Claude Code, GitHub Copilot, Kiro.
- Customer perception risk if KnackForge has no AI delivery story.
- Architects/Tech Leads spend significant time on requirements/architecture/task breakdown/reviews.

### Differentiator (the moat)
- Not "best AI" (OpenAI wins).
- Not "best coding agent" (Cursor wins).
- **Yes: KnackForge Delivery System** — connected requirements + architecture + standards + security + approvals + governance + deployment.
- The moat is the **KnackForge delivery methodology**, not the model.

### First real customer
- Phase 1: Internal KnackForge project (active customer engagement with Jira, GitHub, cooperative team).
- Phase 2: One design partner (trusted customer).
- Phase 3: External offering.

### Pricing
- Phase 1: Internal Accelerator.
- Phase 2: Included within services engagements.
- Phase 3: Commercial packaging evaluation.
- Pricing posture: **undecided for now**, stated explicitly in brief.

### Working mode
- **Coaching path confirmed.** Rationale: building a fundable strategic asset, not a hackathon project.
