The **audit log** is the most important artefact in the platform. It is **append-only**, **cross-account**, and **queryable by `tenant_id`, `run_id`, `stage`, `tool`, `actor`**. The log is the SOC 2 / ISO 27001 control.

## The schema

```typescript
{
  id: string;                  // ULID, monotonically increasing
  tenant_id: string;           // kebab-case stable identifier
  run_id: string;              // ULID, the run this row belongs to
  stage: StageName;            // 'ideation' | 'architect' | 'dev' | 'qa' | 'security' | 'devops' | 'docs'
  tool: string;                // e.g., 'github.create_pr'
  actor: string;               // e.g., 'agent:developer' | 'user:cto@acme-corp'
  input_sha: string;           // sha256 of the input payload
  output_sha: string;          // sha256 of the output payload
  args_hash: string;           // sha256 of the args, for replay
  started_at: string;          // ISO 8601 UTC
  ended_at: string;            // ISO 8601 UTC
  tokens_in: number;           // prompt tokens
  tokens_out: number;          // completion tokens
  usd: number;                 // dollars spent
  result: 'ok' | 'error' | 'paused';
  error?: { code: string; message: string; stack?: string };
  metadata?: Record<string, unknown>;
}
```

The full schema lives in `apps/orchestrator/src/audit/schema.ts`.

## The query surface

The Audit Agent exposes a typed query interface:

```typescript
import { AuditLog } from '@fora/audit';

const log = AuditLog.connect({ region: 'us-east-1', bucket: 'fora-prod-audit-archive' });

// All actions for a tenant in a date range
const rows = await log.query({ tenant_id: 'acme-corp', from: '2026-06-01', to: '2026-06-30' });

// All tool calls for a specific run
const run = await log.getRun('01HXYZ...');

// All secret reads by an actor
const secrets = await log.query({ tenant_id: 'acme-corp', actor: 'agent:developer', tool_prefix: 'secrets.' });
```

The query surface is O(n) over the S3 keys (Athena-backed) and runs in **< 5 s for a 1M-row scan**.

## The cross-account boundary

```
apps/orchestrator ────►  SQS in audit account  ────►  S3 (object lock, 365 d)
   (write only)            (Lambda consumer)             (read by customer)
```

The orchestrator has an IAM role that can **only** write to the audit SQS. It cannot read from the audit S3 bucket. The customer (and the customer's auditor) can read the S3 bucket with their own IAM role.

## The retention

| Tier | Retention | Why |
| --- | --- | --- |
| **Hot** (S3 Standard) | 30 days | Recent runs, fast query |
| **Warm** (S3 Standard-IA) | 90 days | Mid-age, occasional query |
| **Cold** (S3 Glacier Instant Retrieval) | 245 days | Long-tail, audit only |
| **Total** | **365 days** | SOC 2 Type II window |

A customer can request a longer retention (e.g., 7 years for HIPAA) via a per-tenant override in `engagements/<slug>/conventions.md`.

## The immutability rule

The audit log is **immutable**. A customer cannot request a deletion. A tenant offboarding does **not** delete the audit log; the log is retained for the full 365-day window. This is the [SOC 2 Availability + Processing Integrity](https://github.com/fora-platform/fora/blob/main/workspace/customer/standards.md) bar.

A "delete the audit log" customer request is **declined**, not negotiated. The decline is documented in writing and tracked as a closed-lost entry in the deal record.

## The compliance exports

A customer (or their auditor) can request a SOC 2 / ISO 27001 export of the audit log. The export is generated by the Audit Agent and shipped via a customer-encrypted S3 link or via the Forge console (for live exports with date-range filters).

```bash
# example: export a tenant's audit log for SOC 2
fora-cli audit export \
  --tenant=acme-corp \
  --from=2026-01-01 \
  --to=2026-06-30 \
  --format=csv \
  --include='tool,actor,result,started_at,ended_at' \
  --output=acme-corp-soc2-2026h1.csv
```
