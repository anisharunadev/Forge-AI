### Vision override (user steering, post-extraction)

**Vision statement (locked):**
> Forge is a connected delivery intelligence and acceleration platform that transforms delivery knowledge into a reusable organizational asset. By combining Organization Knowledge, Project Intelligence, and governed delivery workflows, Forge ensures every customer engagement inherits the same engineering standards, architectural rigor, security controls, and delivery practices—regardless of team, project, or technology stack.
>
> Forge enables KnackForge to scale delivery quality through systems rather than individual expertise, turning accumulated project experience into a durable competitive advantage.

**Leadership-facing framing (added per user):**
> Forge reduces dependence on individual experts by making delivery knowledge explicit, governed, reusable, and discoverable. As KnackForge grows, delivery quality becomes a property of the system rather than a property of who happens to be assigned to a project.
>
> Explicitly addresses: key-person dependency, inconsistent delivery, scaling teams, knowledge loss.

**One-sentence vision:**
> Forge enables KnackForge to scale delivery quality through systems, not individuals, by capturing, governing, propagating, and reusing delivery knowledge across every customer engagement.

**North Star Metric (CHANGED from brief):**
- ~~Delivery Predictability~~ → **Time to Trusted Delivery (TTTD)**
- **Definition:** Time from approved requirement to deployment-ready release package with all required approvals.
- **Why the shift:** Predictability is a second-order outcome leadership can't see every day. TTTD is observable, daily, and aggregates planning + architecture + development + QA + security + governance efficiency. Predictability becomes a supporting metric.

**Supporting metrics (initial set, refined during Section 2):**
- Predictability (variance)
- Rework Rate
- Story Reopen Rate
- Architecture Review Time
- Onboarding Time
- Project-intelligence metrics (carried from brief: Time to Project Understanding, Architecture Discovery Coverage, Question Resolution Accuracy, Knowledge Acquisition Time)

**Three Pillars (revised):**
1. **Codified Delivery Methodology** — Standards, governance, patterns, and best practices become reusable assets.
2. **Knowledge Propagation** — Delivery knowledge flows forward through the SDLC as typed artifacts and approvals. Uses brief's vocabulary. Chain: Requirements → Architecture → Development → Security → Deployment.
3. **Delivery Intelligence** — Every engagement contributes reusable knowledge that accelerates future work. Umbrella covering institutional memory, project intelligence, refactor intelligence, cross-project learning, knowledge graphs.

### Success Metrics (Section 2) — user steering after first draft

**TTTD Definition (tightened):**
> Elapsed time from an approved requirement entering the delivery system to the creation of a deployment-approved release package containing all required artifacts, approvals, security validations, quality validations, and deployment instructions required for production release. TTTD measures the efficiency of the governed delivery process and intentionally includes all mandatory approval gates.

Rationale for tightening: original definition left "deployment-ready" ambiguous (different teams would interpret differently — e.g., QA done vs. code merged vs. infra provisioned). New definition pre-empts post-hoc arguments about gate-skipping.

**Metric structure (reorganized into three groups):**

- **North Star:** Time to Trusted Delivery (TTTD)

- **Business Outcomes:** Predictability, Rework Rate, Story Reopen Rate, Architecture Review Time, Onboarding Time

- **Project Intelligence Outcomes:** Time to Project Understanding, Architecture Discovery Coverage, Question Resolution Accuracy, Knowledge Reuse Rate (new), Architecture Drift Rate (new)

- **Counter Metrics (anti-gaming):** Architecture Approval Skip Rate, Security Approval Skip Rate, Production Incident Rate, Defect Escape Rate, Knowledge Reuse Avoidance Rate, Human Override Rate (new)

**Additions:**
- **Delivery Knowledge Reuse Rate** — % of stories, ADRs, or projects that reference existing artifacts rather than creating new ones. Strategic; measures whether the knowledge layer is actually working.
- **Architecture Drift Rate** — % of implemented changes that diverge from approved architecture. Powerful executive metric; directly addresses enterprise pain.
- **Human Override Rate** — % of generated artifacts rejected or heavily modified by human reviewers. Trust signal; early warning that output quality or standards alignment has regressed.

**Removals:**
- **Knowledge Acquisition Time** (operational, not strategic). Merged into Time to Project Understanding. Leadership cares about understanding, not ingestion clock-time.

**Pilot framing (verbatim sentence added):**
> Forge will not commit to target improvements until baseline measurements have been established through pilot engagements. Initial pilot success will be measured by successful baseline establishment, user adoption, and statistically significant directional improvement across the defined metrics.

Rationale: protects the initiative from premature numeric commitments; demonstrates rigor to leadership without making promises the pilot data hasn't earned.

## v1.1 Strategic Pivot — Agent Operating System (post-finalization)

**Status:** PRD moved from `final` (v1.0) → `draft` (v1.1) for strategic refresh. Re-marked `final` once refresh complete.

**Pivot summary:**

| v1.0 positioning | v1.1 positioning |
|---|---|
| Forge = **SDLC Agent** (Forge *is* an agent doing SDLC tasks) | Forge = **Agent Operating System** (Forge *orchestrates* external agents) |
| NFR-029: Model Provider Independence (multi-provider LLM) | NFR-029: **Agent Runtime Portability** — Forge is agent-agnostic at runtime |
| Forge chooses models internally; users interact with one Forge agent | Users configure which agent runs which stage (Architecture → Claude Code, Development → Codex, Testing → Gemini, Security → Claude, Deployment → Hermes) |
| Knowledge = Project Intelligence + Org Knowledge Layer | Knowledge = **Knowledge Center** (full knowledge management: docs, repos, Confluence, Jira, Slack, Teams, PDF, ADR, API specs, DB schemas, Figma, AWS) |
| Connectors = Engineering + light others | **Connector Center** — full marketplace (Engineering, PM, Documentation, Cloud, Communication) |
| Single-product positioning | **Control plane positioning** — connect any model, any agent, any connector; visualize entire SDLC + knowledge graph |

**Defensibility upgrade:** A single SDLC agent competes head-on with Claude Code, Codex, Cursor Agent. A control plane that *orchestrates* all of them — providing the governed substrate, the knowledge layer, the audit trail, the visualization, the connectors — is much harder to displace. The differentiator is no longer "the agent" but "the system the agents live in."

**New capability categories added:**

- **Agent Center** (Foundation expansion): configure which external agents run which stages; configure model providers; manage agent assignments.
- **Knowledge Center** (Phase 0 expansion): broader than Project Intelligence — Documentation, Slack, Teams, PDF, Figma, etc., all unified into the Knowledge Graph.
- **Connector Center** (Foundation expansion): marketplace-grade connector management, expanded from V1-required set to a full registry.
- **Realtime Agent Dashboard** (V1-Optional → CEO demo material): live progress of multiple agents running concurrently.
- **Agent Workflow Visualization** (M7+ / V1-Optional): React Flow / Cytoscape-driven live view of agent orchestration.

**NFR-029 upgraded:** Was "Model Provider Independence" (multi-provider LLM abstraction). Now **Agent Runtime Portability** — Forge is portable across agents at runtime. The application never hard-codes a single agent's identity, tool surface, or output shape. This is a stronger invariant than multi-provider-LLM because it covers the *agent* abstraction layer, not just the model layer beneath it.

**V1 packaging reframed:**

- **Package 1 — Forge Foundation:** Organization Knowledge, Connector Center, **Agent Center**, RBAC, Audit, Multi-Tenant
- **Package 2 — Project Intelligence:** Repository Discovery, **CodeGraph**, Knowledge Graph, Service Catalog, Dependency Map, Impact Analysis
- **Package 3 — Architecture Accelerator:** ADR, API Contract, Task Breakdown, Risk Register

**Persona implication:** The Tech Lead becomes an **Agent Orchestrator** — configures which agents run which stages, monitors the Realtime Agent Dashboard, intervenes on escalations. The job-to-be-done narrows but elevates: from "do the SDLC work" to "run the agents that do the SDLC work, governed."

**North Star Metric unchanged:** TTTD remains. The pivot changes *how* TTTD is achieved (via orchestrated agents rather than a single Forge agent), not *what* it measures.

**Strategic posture summary (the user's words):**
> Forge should not force users to use one AI agent. Forge should become the control plane where organizations connect any model, any agent, any connector, and visualize the entire SDLC and knowledge graph in one governed platform.

That sentence is the v1.1 thesis.
