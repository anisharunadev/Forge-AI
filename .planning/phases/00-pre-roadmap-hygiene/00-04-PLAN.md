<objective>
Add a Pydantic v2 `model_validator(mode="after")` to the `Settings` class in `backend/app/core/config.py` that raises `ValueError` when `dev_auth_bypass is True` and `environment != "development"`. The validator fires at `Settings()` instantiation time, and because `settings = get_settings()` is evaluated at module import (line 111 of `config.py`), a misconfiguration causes the process to exit non-zero at import — before FastAPI boots, before any request is served. Cover the guard with three unit tests in a new `backend/tests/test_config.py` and optionally update the stale `FORA_DEV_AUTH_BYPASS` docstring in `backend/app/core/security.py`.

Purpose: Close the PITFALL that lets `DEV_AUTH_BYPASS=1` silently grant a synthetic `dev@forge.local` principal (with `forge:admin` and every `ideation:*` permission) in production. Today the bypass is honored in `get_current_principal` regardless of `environment`. The guard makes the dangerous combination impossible to deploy.

Output: A `Settings` class that refuses to instantiate (and therefore refuses to import) when the bypass is enabled outside dev. Three new unit tests proving the guard fires, allows, and is a no-op when disabled.
</objective>

       - Body: a single `if` block that raises `ValueError` with the message `f"DEV_AUTH_BYPASS=1 is only allowed when ENVIRONMENT=development. Got ENVIRONMENT={self.environment!r}. Refusing to boot."` when `self.dev_auth_bypass and self.environment != "development"`. Otherwise `return self`.

    Why `model_validator(mode="after")` (and not `mode="before"` or `mode="wrap"`): Pydantic v2's `mode="after"` runs after field validation, so `self.dev_auth_bypass` and `self.environment` are already coerced to their declared types (`bool` and `Literal[...]`) by the time the validator executes.     Why a `ValueError` (and not a custom exception): Pydantic v2 wraps any exception raised inside a `model_validator` in a `pydantic_core.ValidationError`. The wrapping is consistent across all validator failures, so the test asserts against `pydantic.ValidationError` (which is `pydantic_core.ValidationError` re-exported) — not against `ValueError` directly. The plan's verify step proves the import-time failure, which is the contract that matters.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| env vars → process boot | `DEV_AUTH_BYPASS` and `ENVIRONMENT` are read by `Settings` at import time. A misconfigured deployment sets the wrong combination and the process refuses to start. This is fail-secure. |
| `get_settings()` cache → test isolation | `@lru_cache` means the validator runs once per process. A test that mutates env vars mid-test must call `get_settings.cache_clear()` or the validator will not re-run with the new env. The test fixture pattern in Task 2 calls `cache_clear()` before each `get_settings()` call. |
| `lifespan` ordering → import order | The validator runs at `import app.core.config`, which `app.main` does at line 22. The validator fires BEFORE the `lifespan` async context manager runs. The fail-fast is guaranteed by import order, not by startup checks. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-04-1 | Elevation of Privilege | `DEV_AUTH_BYPASS=1` in production | mitigate | The Pydantic v2 `model_validator(mode="after")` raises `ValueError` at `Settings()` instantiation. Because `settings = get_settings()` runs at module import (line 111 of `config.py`), the process exits non-zero before any request is served. The validator's message names the misconfigured env vars so the operator can fix the deployment. |
| T-00-04-2 | Denial of Service | validator fires on a legitimate production deploy | accept | A legitimate production deploy will NOT set `DEV_AUTH_BYPASS`. The validator only fires on the dangerous combination. If an operator accidentally sets `DEV_AUTH_BYPASS=1` in a prod env file, the deploy fails — which is the intended behavior. The error message is explicit about how to fix it. |
| T-00-04-3 | Tampering | operator bypasses validator via `lru_cache` reuse | accept | The `@lru_cache` means the validator runs once per process. An operator cannot bypass the validator by setting env vars mid-process (e.g., via `os.environ` mutation) because `get_settings` returns the cached instance. If the process was started with the wrong env, the cache is poisoned — but the process never started. This is fail-secure. |
| T-00-04-4 | Information Disclosure | `ValueError` message reveals env var values | accept | The error message includes `self.environment!r` (the current value of the `environment` field). This is not sensitive — `environment` is a `Literal["development", "staging", "production", "test"]` and the value is something the operator already knows. The message does NOT include `DEV_AUTH_BYPASS` (which is just `True`/`False`). |
</threat_model>
