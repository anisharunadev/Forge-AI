<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI runner ‚Üí native binding | `node-pty` is built by `node-gyp` during `pnpm install`. If the runner lacks `build-essential` (Ubuntu) or Xcode CLT (macOS), the install fails and the workspace does not resolve. |
| apps/forge ‚Üí forge-terminal-server bin | The `dev:terminal` script invokes the bin by name. pnpm resolves the name through the workspace link's `node_modules/.bin/`. If the bin path is wrong, the dev stack breaks silently. |
| package name ‚Üí bin resolution | The bin name `forge-terminal-server` must match the `name` field in the package's `package.json` for pnpm's `bin` linking to work. A typo causes `command not found` at dev-time. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-02-1 | Denial of Service | `node-pty` native build | mitigate | The CI runner is `ubuntu-24.04` per `ci-monorepo.yml`. The runner ships with `python3` and `make`; `build-essential` is NOT preinstalled. The task action explicitly says "run `pnpm install`" ‚Äî if this fails on a fresh CI runner, the executor must add an `apt-get install -y build-essential python3` step to `ci-monorepo.yml` and re-run. This is documented in RESEARCH.md Risks & Landmines (HYG-02, "Native bindings") and in the Assumptions Log (A2). |
| T-00-02-2 | Tampering | `bin` field in `package.json` | mitigate | The bin name in `package.json` (`forge-terminal-server`) and the `dev:terminal` script invocation in `apps/forge/package.json` are tightly coupled. The task action names both explicitly; the verify step greps for both. |
| T-00-02-3 | Information Disclosure | PTY sidecar (no auth) | accept | The sidecar binds to `127.0.0.1` by default and has zero auth. This is a known pre-existing property (documented in the source's leading docstring) and is unchanged by this refactor. The threat is a dev-time concern, not a production concern, because the package is `private: true` and not published. |
| T-00-02-4 | Elevation of Privilege | lockfile churn | mitigate | A PR that changes `apps/forge/package.json` but does NOT regenerate the lockfile will fail CI on `--frozen-lockfile`. The task action explicitly says "Run `pnpm install`" and the verify step checks `git diff pnpm-lock.yaml` is non-empty. |
</threat_model>
