<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI gate → backend code | The gate scans `backend/` for `import litellm`. A future contributor can bypass the gate by writing `import litellm as _litellm` (renaming the module) — the regex matches on the bare name. This is a known limitation; the gate catches the obvious case. |
| CI gate → apps/forge/lib code | The gate scans `apps/forge/lib/` for UUID literals. A future contributor can dodge the gate by writing the UUID as a `Buffer.from([0,0,0,...]).toString()`-style runtime construction. The gate does not catch dynamic construction; it enforces the discipline of "use a config constant, don't inline." |
| apps/forge/lib → apps/forge/config | The dev seeds move from `lib/` to `config/`. The new boundary is "config owns seed/runtime values; lib owns typed client code." A future contributor who adds new UUID literals must understand this split. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-03-1 | Tampering | `import litellm` outside canonical client | mitigate | The CI gate fails the build on any match outside `backend/app/services/litellm_client.py`. The gate is line-anchored (`^\s*import\s+litellm\b`) so it does not match string literals or comments where `import litellm` appears inside a docstring (the triple-quote appears first). False-positive risk: a docstring starting with `import litellm` would match; mitigation: the executor can extend the gate with `grep -v '^\s*"""'` if a real false positive appears. |
| T-00-03-2 | Information Disclosure | UUID literals in `apps/forge/lib/` | mitigate | The two known literals move to `apps/forge/config/dev-seeds.ts`. The gate prevents re-introduction. False-positive risk: a future contributor adds a 36-char hex string that happens to look like a UUID (e.g., a JWT blob) — the gate false-positives. Mitigation: the regex requires 8-4-4-4-12 layout which is the canonical UUID shape; a JWT blob rarely matches. If a real false positive appears, the regex can be tightened to require version+variant nibbles (`4[0-9a-fA-F]{3}-[0-9a-fA-F]{4}-[89ab][0-9a-fA-F]{3}-...`). |
| T-00-03-3 | Elevation of Privilege | grep regex bypass via `import * as litellm` | accept | A future contributor can write `import * as litellm from 'litellm'` — the regex `import\s+litellm\b` matches the bare name after `import * as`, so this is actually caught. The bypass would require `import { chat } from 'litellm'` style, which the regex does NOT match (it has a binding name, not the bare module). This is a known gap; the gate catches the obvious case and a code review catches the rest. |
</threat_model>
