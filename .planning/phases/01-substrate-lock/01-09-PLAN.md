    1. **Header**: `# ADR-009: Cost Ledger Schema and Cumulative Cap`. Metadata block (Status: Accepted, Date: 2026-06-26, Deciders: Forge Architecture Working Group, Consulted: Pilot operations + Cost-attribution working group, Informed: Engineering).

    2. **Context and Problem Statement**: Pilot traffic must attribute every LLM call's cost to a `tenant_id` + `project_id` + `run_id` + `agent` envelope, enforce a per-run cumulative cap before the run starts (PITFALL-2), and reject any LLM call that would exceed the cap. Without a typed ledger, cost attribution is best-effort and the cap is bypass-able.

    3. **Decision Drivers**:
       - Rule 1: provider-agnostic — cost is computed uniformly across providers.
       - Rule 6: auditability — every cost row must link to an audit row.
       - NFR-044: workflow budget — per-run ceiling + spent + remaining surfaced at every gate.

    4. **Considered Options**:
       - Option A: Derive cost from `litellm` response (`response.usage.total_cost`). Pro: zero custom math. Con: provider-dependent field name; LiteLLM Proxy may strip the field; can't compute projected cost pre-call.
       - Option B: Custom `cost_ledger` table with projected + actual rows computed from prompt estimate × model price. Pro: pre-call admission control works; uniform across providers. Con: must maintain a model price catalog.
       - Option C: External cost service (e.g., CloudZero). Pro: rich analytics. Con: pilot doesn't need it yet; adds a vendor.

    5. **Decision**: **Option B**. The cost_ledger table schema is appended below.

    6. **Consequences**:
       - Positive: pre-call admission works (PITFALL-2 closed); uniform attribution; runs UI can show "Run budget: $X / Used: $Y" before the run starts.
       - Negative: must maintain a model price catalog (initial seed: `litellm_model_pricing.yaml` in `backend/app/services/litellm_pricing/`).
       - Neutral: the ledger is append-only — never updated after insert.

    7. **Appendix A — JSON Schema for `cost_ledger` table**:
       ```json
       {
         "$schema": "https://json-schema.org/draft/2020-12/schema",
         "title": "CostLedgerRow",
         "type": "object",
         "required": ["ledger_id", "run_id", "tenant_id", "project_id", "agent", "model", "prompt_tokens", "completion_tokens", "cost_usd", "projected", "recorded_at"],
         "properties": {
           "ledger_id": {"type": "string", "format": "uuid"},
           "run_id": {"type": "string", "format": "uuid"},
           "tenant_id": {"type": "string", "format": "uuid"},
           "project_id": {"type": "string", "format": "uuid"},
           "agent": {"type": "string"},
           "model": {"type": "string"},
           "prompt_tokens": {"type": "integer", "minimum": 0},
           "completion_tokens": {"type": "integer", "minimum": 0},
           "cost_usd": {"type": "number", "minimum": 0},
           "projected": {"type": "boolean"},
           "recorded_at": {"type": "string", "format": "date-time"}
         }
       }
       ```

    8. **Appendix B — Cumulative cap rule**: `sum(cost_usd WHERE run_id = X AND projected = false) <= settings.run_budget_cap_usd[tenant_id]`. The cap is checked before every LLM call in `litellm_client.py` (per plan 01-02).

    1. **Header**: `# ADR-010: Pilot-vs-Multi-Tenant Conflict-Resolution Policy`. Status: Accepted, Date: 2026-06-26.

    2. **Context**: OQ-P3 (conflict volume) and OQ-007 (source-of-truth conflict) need an explicit decision policy. A pilot with one tenant has different conflict-resolution needs than a multi-tenant deployment — the same policy either over-engineers pilot or under-engineers MT.

    3. **Decision Drivers**:
       - Rule 5 (Layer Isolation): Org Knowledge shared within tenant; Project Intelligence isolated per project.
       - Rule 2 (Multi-tenancy): every conflict decision must carry `tenant_id` + `project_id`.
       - Pilot pragmatism: ship a working conflict policy in Phase 1 without over-building for tenants #2..N.

    4. **Considered Options**:
       - Option A: Uniform policy across pilot + MT. Pro: no migration later. Con: over-engineers pilot.
       - Option B: Pilot-vs-MT dual policy with documented escalation path. Pro: ship pilot fast, swap policy at tenant #2. Con: two policies to maintain.
       - Option C: Defer all conflict decisions to Phase 3. Pro: no policy now. Con: blocks Phase 2 from surfacing conflicts.

    5. **Decision**: **Option B**.

    6. **Consequences**: Per-tenant policy override is wired through `settings.conflict_policy_overrides: dict[UUID, str]` (UUID = tenant_id, value = `"pilot"` or `"mt"`). Default = `"pilot"`.

    7. **Appendix A — Decision Table**:

       | conflict_type | pilot_policy | mt_policy | escalation_path |
       |---------------|--------------|-----------|-----------------|
       | architecture_overlap | last-write-wins + audit row | per-tenant CRDT merge | tenant admin |
       | security_conflict | deny + alert | deny + per-tenant quarantine | security lead |
       | deployment_conflict | deny + alert | deny + per-tenant rollback window | SRE oncall |
       | cost_cap_exceeded | deny + retry-after-1h | deny + per-tenant backoff | cost owner |
       | schema_drift | alert + auto-migrate | alert + manual review | data steward |

    8. **Appendix B — Pilot cutoff rule**: A pilot deployment switches to `mt_policy` when `tenant_count >= 2`. Until then, `pilot_policy` applies regardless of the per-tenant override (logged as a warning).
