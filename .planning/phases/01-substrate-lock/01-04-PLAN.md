<objective>
Implement an `APPROVAL_EXPIRED` cron scheduler that fires for runs past their per-phase/per-tenant timeout, emit an `APPROVAL_EXPIRED` event through the existing event bus, and surface a "Stale approval" badge in the runs UI (closing PITFALL-6).

Purpose: PITFALL-6 is the silent-approval gap — today an approval can sit for 24+ hours with no scheduler-side signal, and the runs page has no visual marker. This plan wires the scheduler, makes the timeout configurable per phase and per tenant, and ships the badge.

Output: A cron-driven `approval_expiry.py` scheduler that scans pending approvals every minute, publishes `APPROVAL_EXPIRED` events, plus a `<StaleApprovalBadge>` React component bound to the runs page.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Scheduler → approval_log | Reads pending approvals; must not write without recording who decided (the scheduler records `decided_at=now`, `granted=False`, `reason="timeout"`). |
| Scheduler → event_bus | Publishes APPROVAL_EXPIRED; must carry tenant_id + project_id envelopes (Rule 2). |
| Frontend → approval.expiresAt | UI trusts the server-computed `expires_at`. Client-side date math is for visual rendering only. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04-1 | Denial of Service | Scheduler loop | mitigate | `scan_once` is bounded by the number of pending approvals (small in practice); interval is 60s; no recursion. |
| T-01-04-2 | Tampering | Tenant override config | mitigate | `_validate_timeout_overrides_positive` model_validator enforces `[1, 168]` bounds at Settings instantiation. Negative or zero values raise `ValueError` at boot. |
| T-01-04-3 | Repudiation | Expired approval record | mitigate | The scheduler writes `decided_at`, `granted=False`, `reason="timeout"` to the row — the audit log has the full provenance of who/what/when timed out the approval. |
| T-01-04-4 | Spoofing | Frontend expiresAt | accept | The server is the source of truth for `expires_at`. Client math is visual only. |
| T-01-04-5 | Information Disclosure | Per-tenant timeout config | accept | The dict lives in `Settings` (in-memory only); not exposed via API. |
| T-01-04-SC | Tampering | npm/pip/cargo installs | mitigate | slopcheck + blocking human checkpoint for [ASSUMED]/[SUS] packages; APScheduler must be verified as already in `requirements.txt`. If absent, add a `<task type="checkpoint:human-verify">` to verify via pypi.org before install. |
