# Forge 0.8.6 — Memory system integration (memory.list + memory.retrieve_file over the resolver)

**Owner:** CTO (temporary; per Forge 103 hiring decision)
**Status:** done
**Acceptance:** [Forge 413](/FORA/issues/FORA-413) (sub-goal 0.8.6 of [Forge 103](/FORA/issues/FORA-103))
**Blocked by:** [Forge 409](/FORA/issues/FORA-409) (0.8.3 materialization, shipped 2026-06-20, commit `b1119c85`) and [Forge 411](/FORA/issues/FORA-411) (0.8.4 extension hook, shipped 2026-06-20, commit `1b295638`).

## What this ships

The orchestrator's read path for workspace files. Two new MCP tools on the
Memory MCP server, both thin adapters over the existing 0.4 retrieval
contract (`agents.workspace_resolve`):

- `memory.list({tenant | tenant_id, namespaces?})` — walks the tenant's
  merged seed + override tree via the resolver and returns one entry per
  file with `{relpath, source, path, size, mtime_ns, seed_path}`.
  `source` is `"tenant"` when the tenant override or extension wins,
  `"seed"` when only the seed has the file.
- `memory.retrieve_file({tenant | tenant_id, path | relpath})` — resolves
  one file and returns its content + metadata. Tenant override wins; seed
  is the fallthrough; `source=null` and `found=false` when neither side
  has the file.

No re-implementation of memory storage. No new transitive dependencies.
The SQLite fact store (`MemoryStore.recall`) is unchanged; the new tools
live next to it as the file-content read path.

## Before / after — orchestrator read path

**Before** (0.4 retrieval contract; pre-FORA-409 / 0.8.3 / 0.8.4):

```text
Orchestrator -> Memory MCP -> recall / retrieve       (SQLite facts only)
Orchestrator -> direct read of workspace/<rel>        (seed only; ignores tenants/)
```

The orchestrator could recall facts from the SQLite store (which was
seeded once from `workspace/*`), but every direct file read bypassed the
tenant contract — a customer override at `tenants/acme/workspace/customer/standards.md`
was invisible to any sub-agent that opened `workspace/customer/standards.md`
directly. Tenant-specific conventions and standards simply did not load.

**After** (0.8.6):

```text
Orchestrator -> Memory MCP -> list({tenant})                 -> discover tree
Orchestrator -> Memory MCP -> retrieve_file({tenant, path})   -> read content
                                                   (resolver picks tenant override or seed)
Orchestrator -> Memory MCP -> recall / retrieve               (facts; unchanged)
Orchestrator -> resolve_stage_files(stage, slug=tenant)       -> tenant-aware path list
                                                   (per FORA-411 / 0.8.4)
```

The `list` + `retrieve_file` pair is the new "orchestrator read path"
the Forge 413 issue body calls for. The orchestrator calls `list` once to
discover the resolved tree (with `source` labels so it can see shadows
and extensions), then `retrieve_file` for each file it wants. Tenant
overrides win transparently; the orchestrator never touches `tenants/<slug>/`
directly. `resolve_stage_files` (FORA-411) is still available for the
stage-injection flow that pre-computes a per-stage list of paths.

## Acceptance — Forge 103 AC #4 (in `FORA-413`)

> Smoke test: tenant `acme` with one override at `customer/standards.md`;
> `memory.list(tenant=acme)` shows the resolved tree;
> `memory.retrieve(tenant=acme, customer/standards.md)` returns the tenant
> version; same call with `tenant=globex` returns the seed.

The smoke transcript below captures the full run end-to-end (FORA-32
regression suite + Forge 413 AC #4). Every existing assertion still
passes; the new AC #4 cases are:

```text
[OK  ] list(acme) - 22 files; customer/standards.md source=tenant, seed_path=standards.md
[OK  ] retrieve_file(acme, customer/standards.md) - source=tenant, 414 bytes (override)
[OK  ] retrieve_file(globex, customer/standards.md) - source=tenant, 10535 bytes (seed-equivalent)
[OK  ] list(globex).customer/standards.md - source=tenant, size=seed-equivalent
[OK  ] list rejects invalid slug - invalid_slug error surfaced
```

The acme vs globex distinction is content identity, not the `source`
label. Both tenants are fully materialized (FORA-409), so on-disk both
files exist on the tenant side and the resolver reports `source=tenant`
for both. The acceptance contract is "the call returns the seed
[content]" — `globex`'s `customer/standards.md` is byte-identical to the
seed (10535 bytes raw), while `acme`'s is the 414-byte ACME override.
The smoke asserts both halves: globex's bytes match the seed exactly,
and globex does NOT return acme's override. The `list` cross-check on
the `size` field confirms the same fact from the listing side.

## Tool surface

`make_server()` now registers 13 tools (was 11). The two new ones:

| Tool | Signature | Returns |
|------|-----------|---------|
| `list` | `{tenant \| tenant_id, namespaces?, max_depth?, max_files?, actor?}` | `{tenant, namespaces, files: [{relpath, source, path, size, mtime_ns, seed_path}, …], count, truncated}` |
| `retrieve_file` | `{tenant \| tenant_id, path \| relpath, encoding?, include_content?, actor?}` | `{tenant, relpath, source, path, size, mtime_ns, content, found}` |

Both accept `tenant` (per the Forge 413 spec) OR `tenant_id` (the
existing Memory MCP convention) so the orchestrator can use either
naming. Both go through `agents.workspace_resolve.resolve` /
`read_text` so the path-traversal + slug-grammar invariants carry over
without re-implementation; a bad slug surfaces as `McpError -32602
invalid slug` (verified in the smoke).

Both tools also write a row to the JSONL audit mirror with
`operation="list"` or `operation="retrieve_file"`, `target={tenant,
relpath|source}`, and a `tokens_out` count so the Audit agent (FORA-210)
can sample one stream and see the new file-content read path the same
way it sees fact writes.

## Risk and rollback

- **Risk surface** is the MCP server's tool surface (+2 entries). The
  underlying resolver is unchanged from FORA-411; this ship only adds
  thin adapters and registers them.
- **Rollback**: revert this single commit. The 11 pre-existing tools
  (write, propose, recall, retrieve, inject_for_stage, seed_workspace,
  promote, forget, stats, read_audit, injection_table) are unaffected —
  the tool-list ordering and JSON-RPC wire format are additive only.
- **Open follow-up**: the FORA-413 body names the 0.4 memory owner
  (CEO) as the eventual permanent assignee once 0.8.3 + 0.8.4 land.
  Both have landed. CTO has driven the wiring; the next 0.8.x sub-goal
  (likely 0.8.7 — memory audit + cross-tenant promotion path) is a
  natural handoff to a dedicated memory agent. Recorded in the ship
  comment on FORA-568.
