## What this ships

`agents/workspace_resolve/` — a pure-Python, cache-friendly resolver that
is the single read path the orchestrator and the memory MCP use after a
tenant has been materialized. The contract is:

```text
tenants/<slug>/workspace/<rel>   →  tenant override (if present)
<seed>/<rel>                     →  seed (fallthrough)
```

A tenant can shadow any seed file under `customer/` or `project/`, and
can add new files under those namespaces. A tenant cannot write to
`memory/` — that namespace is platform read-only by contract
(`workspace/README.md §6`) and a write attempt is refused with an audit
event. The CLI is the smallest credible shape for the spec:

```text
python -m agents.workspace_resolve --tenant <slug> --path <rel>
python -m agents.workspace_resolve --tenant <slug> --path <rel> --read
python -m agents.workspace_resolve --tenant <slug> --path <rel> --write --body "..."
python -m agents.workspace_resolve --tenant <slug> --path <rel> --write --body-file <f>
python -m agents.workspace_resolve --clear-cache
```

Exit codes: `0` resolved/written/cache-cleared, `1` not found,
`2` bad input / denied write, `3` partial I/O error. Pure stdlib for the
hot path; the CLI lazy-imports nothing.

## What "resolve" means

After 0.8.3, every materialized tenant has a per-tenant tree under
`tenants/<slug>/workspace/`. The resolver is the single read path that
consults the tenant tree first and falls through to the seed. The
post-condition the orchestrator (and any future sub-agent runtime)
needs: a `(slug, rel)` lookup is a deterministic O(1) on the hot path
and always returns the right file — tenant override, seed, or `None`
if neither side has it.

The resolver also owns the write side of the contract. `write_to_tenant`
gates the write against `PROTECTED_RELPATH_PREFIXES = ("memory/",)`. A
write to `memory/` is refused, an audit row is appended, and no file is
written or modified. The audit log carries the actor, the slug, the
relpath, the bytes attempted, and a stable `reason` token so the Audit
agent (FORA-210) can grep for `reason=memory_tenant_write_blocked` and
treat every hit as a contract violation.

## Design choices surfaced for review

1. **Pure-Python, no subprocess shell-outs.** `os.path.join` + `os.stat`
   + `open` satisfy this. The CLI / Python API is in-process; no
   shell, no `subprocess.run`. The memory_mcp seed path (FORA-409)
   is unchanged; the new resolver is a sibling module.

2. **Process-local LRU cache keyed on `(slug, relpath)`, invalidated
   by mtime drift.** The hot path is one dict lookup + one `os.stat`
   per cache miss per pair. A tenant write or seed write that changes
   a file's mtime invalidates the entry on the next read; no operator
   action is required. The cache is bounded at 4096 entries; LRU
   eviction drops the oldest by insertion order. Thread-safe (single
   `threading.Lock` around the dict).

3. **Negative cache for misses.** A sub-agent loop that asks for the
   same missing file in a hot path does NOT redo the stat on every
   iteration; the cache holds a short negative entry. Future tuning
   can add TTL; the current cache is mtime-driven and matches the
   FORA-410 invariant (a file that exists changes mtime; a file that
   is deleted fails the next stat).

4. **Slug + relpath validation up front, before any I/O.** The slug
   grammar mirrors the platform's tenant_id shape
   (`^[a-z0-9][a-z0-9_-]{0,62}$`). The relpath validator rejects
   absolute paths, parent-traversal (`..` segments), and leading
   `./`. A bad slug → `ResolverError` before any I/O; a bad relpath →
   same.

5. **The seed is NEVER read directly once a tenant is materialized.**
   The orchestrator (and any new runtime caller) must go through
   `resolve()` so the override surface is honored. The pre-existing
   `memory_mcp/injection.get_stage()` API is unchanged; a sibling
   `resolve_stage_files(stage, slug=...)` returns the tenant-resolved
   file paths for the stage's injection table so the orchestrator's
   read path can adopt the resolver without rewriting its table.

6. **`write_to_tenant` is the ONLY sanctioned write path for the
   tenant tree.** It enforces the `memory/` block, refreshes the
   cache for the key, and appends the audit row. Direct edits to
   `tenants/<slug>/workspace/` are not blocked at the filesystem
   level — that's an operator-privilege question, not a library
   question — but every audit row records the actor so any change
   outside `write_to_tenant` is still visible to the Audit agent as
   a missing audit row for an expected event.

7. **No new dependencies.** `agents/workspace_resolve/resolver.py`
   uses stdlib only (`os`, `re`, `json`, `time`, `hashlib`,
   `threading`, `dataclasses`, `typing`). The CLI adds `argparse`
   + `sys`. No third-party imports.

## Critical invariants (recap)

The FORA-409 / FORA-410 / FORA-412 invariants carry over:

- The resolver NEVER writes into the seed tree. A successful
  `write_to_tenant` only touches `tenants/<slug>/workspace/<rel>`.
- The resolver NEVER reads another tenant's tree. The slug grammar
  is the first line of defense; the `os.path.join` walk against
  `tenants_root/<slug>/workspace/` is the second.
- The resolver NEVER crosses the memory/ read-only boundary. A
  write is refused at the door with an audit row, not silently
  dropped.
- The seed remains the source of truth for `memory/`. Tenants get a
  materialized copy (FORA-409) that is byte-identical to the seed
  at materialization time and is NEVER updated by the resolver; the
  only way to evolve `memory/` is via the migrate runner (FORA-412
  / 0.8.5) which writes the seed.

The resolver also leaves the migration runner's protected-path
invariant intact: `_iter_tenant_override_paths` in
`agents/workspace/migrate.py` continues to surface the same
`tenants/<slug>/workspace/<rel>` shadows it always did — the
resolver doesn't introduce a new path convention, it consumes the
one the migration runner already emits.

## What this does NOT do (deferred)

- **Cross-region resolution.** A multi-region deployment would need
  a `tenants_root` that points at the regional mount. The resolver
  takes `--tenants-root` and `$FORA_TENANTS_ROOT` as overrides, so
  the regional wiring is a future flag away.
- **Hot-reload of the cache across processes.** The cache is
  process-local; two processes holding a stale cache will diverge
  on the next mtime change (correctly — the next read refreshes).
  A cross-process invalidation channel (NATS / Redis pub-sub) is a
  v0.5 conversation; the current contract is "the resolver is
  eventually consistent at the mtime boundary."
- **Bulk tenant extension.** `write_to_tenant` writes one file at a
  time. A `bulk_write` that takes a manifest is a v0.2 convenience;
  the existing API composes with a loop and the audit log carries
  one row per call so the trail is intact.
- **Tenant extension of the manifest itself.** A tenant cannot
  override `workspace-manifest.json` (FORA-410) — the resolver
  rejects any relpath starting with `.` or containing `..`, and
  the manifest lives under the seed, not the tenant tree. A future
  `customer-manifest.json` is a v0.5 conversation tied to the
  customer-conventions.md work.

## Risk + rollback

Risk: low. The resolver is opt-in: callers that do not import
`agents.workspace_resolve` see no change. The orchestrator read path
wiring is a one-function addition to `memory_mcp/injection.py`; the
legacy `get_stage()` API is unchanged and any caller that does not
pass `slug=` continues to see the seed directly. The `memory/` write
block is enforced inside `write_to_tenant`; no caller is forced
through the new gate.

To revert this change in an emergency: delete
`agents/workspace_resolve/`, drop the `resolve_stage_files` helper
from `agents/memory_mcp/injection.py`, and revert the `scripts/smoke.sh`
gate. No data on customer systems is touched; the materialized
tenant tree is still read by the legacy code path
(`tenants/<slug>/workspace/<rel>` is the directory the materializer
writes, and any caller that walks it directly continues to work).
