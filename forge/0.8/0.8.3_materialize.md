# FORA-409 / 0.8.3 — Per-tenant workspace materialization

**Owner:** CTO (temporary; per FORA-103 hiring decision)
**Status:** done
**Acceptance:** [FORA-409](/FORA/issues/FORA-409)
**Blocked by:** [FORA-38](/FORA/issues/FORA-38) — 0.7 foundation (shipped 2026-06-19)

## What this ships

`agents/workspace_materialize/` — a pure-Python materializer that
cold-starts a tenant by copying the seed workspace into
`tenants/<slug>/workspace/` and priming the 0.4 memory index for
that tenant. The CLI is the smallest credible shape for the spec:

```text
python -m agents.workspace_materialize --tenant <slug>
python -m agents.workspace_materialize --tenant <slug> --no-prime-memory
python -m agents.workspace_materialize \
    --tenant <slug> \
    --seed-root /opt/fora/workspace \
    --tenants-root /opt/fora/tenants \
    --memory-db /opt/fora/var/memory.db
```

Exit codes: `0` success, `1` partial write / unexpected, `2` bad input.
Pure stdlib for the file-copy path; the memory prime lazy-imports
`agents.memory_mcp.store.MemoryStore` (sqlite + sqlite-vec) only when
`--prime-memory` is set.

## What "materialize" means

The seed (`workspace/`) is the platform's source of truth. A new
tenant's first session mint needs that same Knowledge Layer under
its own `tenants/<slug>/workspace/` tree, plus the 0.4 memory index
loaded with `tenant_id=<slug>` on every fact the seed declares.

The post-condition the cold-start path expects: every sub-agent's
next read for tenant `slug` resolves from `tenants/<slug>/workspace/`
and the memory store returns the seed facts scoped to that tenant.

## Design choices surfaced for review

1. **Pure-Python, no subprocess shell-outs.** `shutil.copytree` /
   `shutil.copy2` satisfy this; the `--no-subprocess` rule is
   recorded in the module docstring so the FORA-103 reviewer can
   grep for it. No `subprocess.run`, no `os.system`, no shell `cp -r`.

2. **Naive copy is fine; the memory prime was the bottleneck.** The
   first cut (calling `MemoryStore.write` once per fact) was
   ~200s for a 50-file seed (~65 fsyncs × 3s each), blowing the
   60s bar. `agents/workspace_materialize/memory_seed.bulk_seed`
   collapses the same writes into one `BEGIN ... COMMIT`; one fsync.
   The audit row invariant (one audit row per fact) is preserved.

3. **The materialize path owns the file copy; the memory path owns
   the audit row.** The CLI writes one audit JSONL row per call
   (`var/materialize-audit.jsonl` by default) carrying the slug,
   file count, byte total, copy + memory timing, materializer
   version, and actor (`workspace-materializer`). Per-fact audit
   rows live in `memory_audit` (the MCP server owns that table).

4. **Slug grammar mirrors the platform's tenant_id shape.** The
   same regex the identity-broker uses for tenant_id claims
   (`^[a-z0-9][a-z0-9_-]{0,62}$`). The slug becomes part of every
   object-prefix key and audit row, so anything that could be a
   path-traversal or shell-injection vector is rejected at the
   door.

5. **Re-running is a clean re-materialization, not a merge.** The
   prior `tenants/<slug>/workspace/` tree is wiped before the
   new copy; a future drift in the seed (a file removed, a file
   renamed) is reflected in the tenant tree on the next call.
   Tenant-owned siblings (`tenants/<slug>/policy.yaml`,
   `tenants/<slug>/cloud_trust.yaml`) live OUTSIDE `workspace/`
   and are untouched.

6. **`bootstrap_if_missing` is the idempotent cold-start entry the
   identity-broker (0.7) calls.** Returns
   `{"status": "bootstrapped", "result": ...}` on miss,
   `{"status": "already_materialized", "tenant_workspace": "..."}`
   on hit. The "have we seen this tenant?" check is the existence
   of `tenants/<slug>/workspace/<sub>/<any_file>` for any of the
   three seed subdirs; the README check would be brittle against
   a future sparse fixture.

## Files

- `agents/workspace_materialize/__init__.py` — public surface
  (`materialize`, `bootstrap_tenant`, `bootstrap_if_missing`,
  `MaterializeResult`, `MaterializeError`, `DEFAULT_AUDIT_LOG`).
- `agents/workspace_materialize/materialize.py` — file copy +
  slug validation + audit row. ~480 lines, pure stdlib.
- `agents/workspace_materialize/memory_seed.py` — batched memory
  prime (one transaction). ~220 lines. Lazy-imports the embedder.
- `agents/workspace_materialize/cold_start.py` — 0.7 hand-off
  surface. Two functions: `bootstrap_tenant` (always materialize)
  and `bootstrap_if_missing` (idempotent variant).
- `agents/workspace_materialize/cli.py` — `argparse` + JSON
  stdout. Composes with shell pipelines and a future HTTP trigger.
- `agents/workspace_materialize/__main__.py` — `python -m`
  shim so the CLI module is also importable as a library.
- `agents/workspace_materialize/smoke_test.py` — 7 ACs:
  materialize <60s, tenant-tree reads, memory index primed,
  tenant-scoped recall, idempotent bootstrap, audit row,
  CLI exit code.
- `scripts/smoke.sh` — adds the `workspace materialize` gate
  after the existing `workspace migrate` gate.
- `forge/0.8/0.8.3_materialize.md` — this evidence file.

## Critical invariants (recap)

The FORA-410 / FORA-412 invariants carry over. The materializer
NEVER touches `engagements/` and NEVER reads/writes/deletes files
in another tenant's `tenants/<other>/workspace/`. The slug grammar
is the first line of defense; the `os.path.relpath` walk against
`seed_root` is the second. A bad slug → `MaterializeError` before
any I/O.

The materializer also leaves the seed alone: it only READS
`workspace/` and WRITES `tenants/<slug>/workspace/`. The seed
manifest (`workspace/workspace-manifest.json`) is not modified.

## What this does NOT do (deferred)

- **Multi-tenant bulk materialize.** A single call materializes one
  tenant. A bulk driver (`--from-tenant-list <file>`) is a v0.2
  convenience. The existing CLI composes with `xargs` / `parallel`
  for the bulk case; the audit log carries one row per call so
  the trail is intact.
- **Incremental update vs. wipe-and-copy.** A future seed update
  to a single file would currently wipe + recopy the whole tenant
  tree. The wipe is milliseconds for the real seed (18 files,
  174 KB) and is the simplest safe semantics; an incremental mode
  that diffs the manifest before copying is a v0.5 conversation
  tied to the version train's tag history.
- **Tenant-owned seed overrides.** A tenant cannot currently
  override a single seed file; the entire tree is wiped on every
  materialize. The override story is the next sub-goal in the
  Knowledge Layer track (the "tenant overrides" half of the
  version train's protected-path invariant).
- **Cross-region materialization.** The CLI takes a `--memory-db`
  but does not coordinate with a remote memory_mcp. A multi-region
  deployment wires this through a future HTTP trigger; the audit
  log row carries enough provenance for the cross-region
  reconciliation.

## Risk + rollback

Risk: low. The materializer is opt-in (the broker calls it
explicitly; nothing in the runtime touches `tenants/` until the
broker does). The wipe-and-re-copy is destructive only of
`tenants/<slug>/workspace/`, never of the tenant-owned siblings
(`policy.yaml`, `cloud_trust.yaml`). A bad materialize is
recoverable by re-running with the same slug — the prior state is
gone but the seed is the source of truth.

To revert this change in an emergency: delete
`agents/workspace_materialize/` and revert the `scripts/smoke.sh`
gate. No data on customer systems is touched.
