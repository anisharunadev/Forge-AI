## 1. Context

The Board's free-form response on FORA-62 interaction `4c94c158-…` asked for
bidirectional Paperclip ↔ Jira ↔ GitHub Issues ↔ "clipup" sync. The
`ask_user_questions` round on FORA-199 interaction `02d96f8c-…` resolved
**Q-`clipup` = `clickup`** (2026-06-17). FORA-202 is the Epic 11.2c
vertical slice for the ClickUp adapter, mirroring FORA-200 (Jira) and
FORA-201 (GitHub). This plan scopes the day-one deliverable.

The ClickUp MCP skeleton (11.2c.1) already ships: 8 typed tools
(`list_tasks`, `search_tasks`, `get_task`, `list_comments`,
`create_task`, `update_task`, `set_task_status`, `add_comment`) over
MCP/stdio, pinned to a single List, mutations gated by
`confirm: z.literal(true)`, mock-backed smoke green (27 assertions
across 11 HTTP requests; scope-guard and mutation-gate verified).
This plan extends 11.2c.1 → 11.2c.5 — the wire-up to the Sync Plane
spine, the idempotency spine, the audit emitter, the mirrors, and the
divergence hook.

---

## 2. Decision (one-line)

Ship the same v0.1 shape as FORA-200 with ClickUp semantics: an
`@fora/clickup-adapter` TypeScript package, an idempotency spine keyed
on `(tenant_id, external_id, op_kind)` against the existing `sync_op`
table (migration 0008), a closed 6-event audit emitter, three mirrors
(issue / comment / status), a webhook receiver, and a divergence-cron
hook. All writes route through the customer-cloud-broker (FORA-126);
all reads use the broker too once the per-service dispatch ships (Phase 2).

### 3.1 — `FORA-407` — Idempotency spine + audit emitter (mirror of FORA-401)

**Owner:** IntegrationEngineer.
**blockedBy:** none (ships first).
**Output:** `apps/clickup-adapter/src/{idempotency,audit,pool_executor,index}.ts`,
migration `0009_clickup_adapter.sql` (ClickUp variant of `sync_op`
table — schema-identical to Jira's row, no new migration needed if
we share the table).

**Why a shared `sync_op` table:** FORA-200's migration 0008 already
declares the `sync_op` table with the closed `op_kind` CHECK constraint
(`issue.create | issue.update | comment.create | comment.update |
stage.transition | webhook.received`). The ClickUp adapter reuses the
table but emits a ClickUp-flavoured `op_kind` set:

```
op_kind = 'task.create' | 'task.update' | 'task.set_status'
        | 'comment.create' | 'comment.update' | 'webhook.received'
```

This requires a **forward migration to widen the CHECK** (Jira's
op_kinds become a subset). Migration `0009_clickup_adapter.sql` does
`ALTER TABLE sync_op DROP CONSTRAINT sync_op_op_kind_check` and
re-adds with the union. The `external_id` and `(tenant_id, external_id,
op_kind)` index is unchanged.

**Acceptance bar:**
- `claim()` property test replays 10× the same key and asserts
  `firstTime: true` exactly once (mirrors FORA-401).
- The first successful claim emits all 6 closed event types
  (`sync.source.{task,comment,status}.ok` + `sync.target.{...}`).
- Replays emit 0 audit events.
- All 6 emit-on-success audit events observed in the audit collector.

**Verification:** `npm --prefix apps/clickup-adapter run test` green
(property test + 6-event assertion). Migration applied locally.

### 3.2 — `FORA-408` — Outbound issue mirror (mirror of FORA-402)

**Owner:** IntegrationEngineer.
**blockedBy:** [FORA-407].
**Output:** `apps/clickup-adapter/src/task-mirror.ts`, the broker-routed
write path.

**ClickUp semantic differences from Jira (per `mcp-servers/clickup/docs/assumptions-vs-jira-mcp.md`):**
- ClickUp has no transition primitive. `task.update({ status })`
  sets status by name; the List's workflow auto-decides if the move is
  legal. The mirror calls `set_task_status` (broker-mediated) with the
  translated status name from §4 ownership table.
- ClickUp plain-text body round-trips losslessly (Jira ADF round-trip
  is lossy). The mirror writes plain text.
- Status translation lives in
  `apps/clickup-adapter/src/status-translation.ts` (new file) and
  reads from `forge/sync-plane/tenants/<slug>/clickup-status-map.yaml`.

**Acceptance bar:**
- `mirror.issue.create()` happy path: `claim()` → broker → ClickUp
  `create_task` → store target ref → emit `sync.source.task.ok`.
- Idempotent on `(tenant_id, clickup_task_id, 'task.create')` replay.
- Foreign-List scope error (`ListScopeError`) is NOT a 5xx; emitted as
  `sync.source.task.fail` with `outcome: 'fail'`, `reason: 'scope'`.

**Verification:** contract test mocks ClickUp, asserts one
`create_task` call, one `set_task_status` call, two
`sync.source.task.ok` events.

### 3.3 — `FORA-409` — Inbound webhook ingest (mirror of FORA-403)

**Owner:** IntegrationEngineer.
**blockedBy:** [FORA-407].
**Output:** `apps/clickup-adapter/src/webhook.ts`, the per-tenant webhook
receiver that lands into `sync.inbox` JetStream subject.

**ClickUp webhook security:**
- ClickUp signs events with `X-Signature: sha256=<hmac-hex>` using
  the per-tenant webhook secret (HMAC-SHA256 over the raw body,
  same as FORA-161 IdP revoke webhook — same code path).
- Each tenant gets its own secret stored in `@fora/secrets-mcp` under
  the `clickup.webhook_secret.<tenant>` key (FORA-128 secrets-mcp
  shipped; pattern already used by FORA-161).
- The receiver MUST verify the HMAC before parsing the JSON; a failed
  HMAC returns 401 and emits `sync.target.webhook.fail` with
  `reason: 'bad_signature'`.
- ClickUp sends a `challenge` ping on subscription creation; the
  receiver echoes it back as the response body (per ClickUp docs).

**Acceptance bar:**
- Webhook contract test: 3 events (taskCreated, taskUpdated,
  taskStatusUpdated) → 3 `SyncEvent`s → 3 `sync.target.webhook.ok`.
- Bad HMAC → 401, zero events emitted.
- `challenge` ping → 200, body = challenge, zero events emitted.
- Idempotency: replaying the same webhook `event_id` lands exactly
  once (via `claim()` against `webhook.received`).

**Verification:** `npm --prefix apps/clickup-adapter run test:webhook`
green (3 events + 2 negative paths + 1 idempotency replay).

### 3.4 — `FORA-410` — Comment mirror (mirror of FORA-404)

**Owner:** IntegrationEngineer.
**blockedBy:** [FORA-407, FORA-409] (need both directions of the wire).
**Output:** `apps/clickup-adapter/src/comment-mirror.ts`.

**ClickUp comment semantics:**
- ClickUp comments are plain text (per `mcp-servers/clickup/docs/assumptions-vs-jira-mcp.md`
  §5). The comment envelope's `body_md` field round-trips unchanged.
- ClickUp supports `notify_all=true` query param — mirror uses
  `notifyAll: false` by default to avoid spamming watchers.
- The `add_comment` mutation is broker-mediated and gated by
  `confirm: z.literal(true)`. No bypass.

**Acceptance bar:**
- Outbound: Paperclip `comment.created` event → `claim()` →
  broker → ClickUp `add_comment` → store target ref → emit
  `sync.source.comment.ok`. Idempotent on `(tenant_id, comment_id,
  'comment.create')` replay.
- Inbound: ClickUp webhook `commentPosted` → envelope with
  `remote_refs.clickup` populated → `sync.target.comment.ok`.

**Verification:** contract test mirroring FORA-404 (5 assertions
including the lossless plain-text round-trip).

### 3.5 — `FORA-411` — Status mirror + divergence job (mirror of FORA-405 + FORA-406)

**Owner:** IntegrationEngineer.
**blockedBy:** [FORA-408, FORA-410].
**Output:** `apps/clickup-adapter/src/status-mirror.ts` and
`apps/clickup-adapter/src/divergence-cron.ts`.

**Status model divergence:**
- ClickUp's `issue.status` ownership row in
  `forge/sync-plane/src/ownership.ts` is `creator` mode with
  `translated_mirror_state`. The creator platform owns status; the
  other platform stores `mirror_state.status` (translated via the
  status map).
- Jira's `transition_issue` is first-class; ClickUp's
  `set_task_status({ taskId, status })` is a plain update. The
  status mirror ALWAYS calls `set_task_status` and trusts the List's
  workflow to accept or reject (a 400 from ClickUp surfaces as
  `sync.source.task.fail` with `reason: 'transition_rejected'`).

**Divergence cron:**
- Hourly `setInterval` job reads `DivergenceQueue.list()` from
  `@fora/sync-plane` and surfaces parked ClickUp events to the
  workbench (`forge/sync-plane/divergence-queue.ts` already exists —
  this just wires the consumer).
- Emits `sync.event.divergence_detected` per parked event.

**Acceptance bar:**
- `mirror.status.transition` happy path: `claim()` →
  `set_task_status` (broker) → `sync.source.task.set_status.ok`
  (a 7th event type, closed-set extension).
- Parked-divergence test: 2 events with conflicting non-empty
  body → `parked` outcome → workbench row visible.
- Cron fires every 60s in test (overridable).

**Verification:** `npm --prefix apps/clickup-adapter run test:status`
green + divergence property test (park-3-events → 1 workbench row).

## 4. Domain lens commitments

| Lens | Day-one contract |
|------|------------------|
| **Idempotency** | Every adapter op runs through `claim()` against the shared `sync_op` table with `(tenant_id, clickup_<id>, op_kind)`. Replay returns `firstTime: false` and emits 0 audit events. |
| **Conflict resolution** | Tier-1 (per ADR-0010 §4) — `clickup.assignee`, `clickup.priority`, `clickup.due_date`, `clickup.list_id` are Tier-1 `single` mode, owner=clickup, `read_only_on_remote`. Tier-2 (title, body, comment.body) — HLC LWW per `@fora/sync-plane/Resolver`. Status — `creator` mode with translated mirror state. |
| **Rate limiting** | ClickUp Cloud: ~100 req/min per token for both reads and writes (per `assumptions-vs-jira-mcp.md` §1). Day-one: reuse `@fora/sync-plane-ratelimit` token bucket + circuit breaker (per FORA-200's pattern). Per-tenant + per-platform queue + composite-edit coalescing for comment storms (per ADR-0010 §7.1 burst-control surface). |
| **Pagination** | Cursor-style via `page` + `pageSize` query params (max 100). The adapter never loads full lists; `list_tasks` is always paged. |
| **Eventual consistency & ordering** | Inbound via ClickUp webhooks (best-effort, 24h redelivery window) + polling backstop every 5 min (per ADR-0010 §3.2). Lag budget: P50 < 30s webhook, P95 < 5min via polling. HLC ordering (Tier-2 fields). |
| **Schema mapping** | `forge/sync-plane/src/ownership.ts` already declares ClickUp fields. Status map per-tenant in `forge/sync-plane/tenants/<slug>/clickup-status-map.yaml`. |
| **OAuth / scopes** | Day-one: ClickUp Personal API Token (account-scoped, not per-token-scoped — per `assumptions-vs-jira-mcp.md` §6 least-privilege). Documented in README: service account scoped to one List. Phase 2 (post-Architect-hire): switch to OAuth 2.0 with explicit `task.read|write|comment.read|comment.write|status.write` scopes. |
| **Webhook vs polling** | Webhook for happy path (ClickUp supports signed events); polling every 5 min as backstop. |
| **Observability** | Every sync op emits a `sync.{source,target}.{task,comment,status,webhook}.{ok,fail}` audit event with `tenant_id, source, target, external_id, op_kind, outcome, actor, claimed_at, metadata`. Stderr logging from the adapter for boot + crash; structured JSON for tool calls. |
| **Backpressure & circuit-breakers** | Reuse `@fora/sync-plane-ratelimit` circuit breaker per platform. Trip on 3 consecutive 5xx; half-open after 30s. |
| **Audit trail & replayability** | Every `claim()` writes a row to `sync_op` with `claimed_by, claimed_at, metadata`. Audit events go to `@fora/db-pool` (FORA-36 forwarder). Replay is reconstructable from the audit log + the ClickUp event_id back-map. |

## 5. Conflict-resolution choices (the lens the ADR-0010 §4 demands)

- **Tier-1 single-owner fields** (per `forge/sync-plane/src/ownership.ts`
  default row 4b):
  - `clickup.assignee` → owner=`clickup`, `read_only_on_remote`.
  - `clickup.priority` → owner=`clickup`, `read_only_on_remote`.
  - `clickup.due_date` → owner=`clickup`, `read_only_on_remote`.
  - `clickup.list_id` → owner=`clickup`, `read_only_on_remote`.
- **Tier-2 HLC LWW fields:** `issue.title`, `issue.body`, `comment.body`
  (writers: all platforms).
- **Status:** `creator` mode with `translated_mirror_state`. The creator
  platform's status wins; the other platforms store a translated
  status name (`clickup-status-map.yaml`). Jira workflow id is
  irrelevant to ClickUp — the translator looks up the ClickUp status
  by name.
- **State precedence (default, tenant-overridable):**
  `paperclip > jira > github > clickup` for state-machine fields only
  (per ADR-0010 §4 last paragraph; FORA-202 inherits, doesn't change).

## 6. Lag budget

| Path | P50 | P95 | Backstop |
|------|-----|-----|----------|
| Paperclip → ClickUp (outbound) | < 5s | < 30s | Retry with exponential backoff (1s, 4s, 16s) on broker-mediated calls; circuit-break at 3 consecutive 5xx |
| ClickUp → Paperclip (inbound webhook) | < 30s | < 2min | Polling backstop every 5min |
| ClickUp → Paperclip (inbound polling) | < 5min | < 6min | Every 5min `updated_after` cursor |
| Reconciliation sweep (full re-read) | < 1h | < 2h | Daily divergence job |

---

## 7. Rate-limit strategy

- ClickUp Cloud: ~100 req/min per token for both reads and writes.
- Per-tenant + per-platform token bucket
  (`@fora/sync-plane-ratelimit`, same package as FORA-200 uses).
- Burst window: 100 req / 60s (1.67 req/s sustained).
- Circuit breaker: trip on 3 consecutive 5xx OR 5 of any 5xx within
  60s; half-open after 30s with 1 probe.
- Comment storm (R-SYNC-03): composite-edit coalescing — N consecutive
  comments from the same Paperclip actor on the same ClickUp task
  within 60s are merged into one outbound edit with an `> _Composite
  of N comments — <range>_` block.

## 8. Idempotency key shape

```
(tenant_id, external_id, op_kind)
```

- `tenant_id` — from broker claim (FORA-163).
- `external_id` — the **Paperclip-issued stable id** (e.g.
  `paperclip:comment/cmt_…` for outbound, `clickup:task/<id>` for
  inbound).
- `op_kind` — closed enum:
  `task.create | task.update | task.set_status | comment.create |
  comment.update | webhook.received`.

The same row in `sync_op` covers replays across the whole 30-day
JetStream retention window; no in-memory state required.

## 9. Observability hooks

- **Boot log** — stderr: `[fora-clickup-adapter] starting — pinned to
  list='<list_id>' (tenant=<slug>), baseUrl='<base_url>'`.
- **Per-call audit event** — 7 closed event types in
  `sync.{source,target}.{task,comment,status,webhook}.{ok,fail}`.
- **Failure events** — `reason` field in `metadata` discriminates
  scope, transition_rejected, bad_signature, rate_limited, circuit_open.
- **Divergence event** — `sync.event.divergence_detected` from
  `DivergenceQueue.list()`.
- **Metrics** — emitted to stdout as JSON for `@fora/agent-runtime` to
  scrape: `{ adapter: 'clickup', op_kind, outcome, latency_ms }`.

## 11. Out of scope (deliberate, per FORA-200 charter)

- **Attachment sync** — non-trivial binary round-trips; same posture
  as FORA-200.
- **Custom-field mapping** — ClickUp custom fields per List; defer to
  Phase 2 with the same shape as FORA-200's custom-field slice.
- **OAuth 2.0** — Phase 2 (post-Architect-hire); day-one ships with
  Personal API Token + documented least-privilege posture.
- **Workspace-wide tokens** — explicitly NOT supported; the MCP and
  the adapter are both List-pinned for safety.
- **Reaction sync** — explicitly local to ClickUp (per ADR-0010 §11).
- **Direct ClickUp writes outside the broker** — never (per FORA-200
  charter); FORA-126 customer-cloud-broker is the only writer.
