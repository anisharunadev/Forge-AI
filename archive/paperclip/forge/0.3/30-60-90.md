## First 30 days — Onboard + earn trust

**Objective:** land inside the codebase, ship one small change, learn the team.

**Deliverables:**

1. **Knowledge-layer orientation pass with CTO** — read [`./charter.md`](./charter.md), [`../0.8/0.8.1_lint.md`](../0.8/0.8.1_lint.md), [`../../HIRING_PLAN.md`](../../HIRING_PLAN.md), the agent-runtime README, and `forge/sync-plane/risk_register.md`. Produce a written "what I learned" doc (≤ 2 pages) under `forge/0.3/onboarding_notes.md`.
2. **First issue** — pick up the **MCP server SDK skeleton + tenant-resolution helper** (a fresh child issue filed at Forge AI-451 close, parented under Forge AI-48). Land it `done` with smoke test, evidence, and PR description. Acceptance: ≥ 1 MCP server (Confluence or Secrets) consumes the new SDK cleanly with no regressions.
3. **Run the smoke matrix** for all 13 production MCP servers against the in-process mock harness; produce a baseline latency / error report under `forge/0.3/slo_baseline_<date>.md`. This becomes the SLO baseline for the day-60 dashboard.
4. **Architecture review** with the Senior Engineer (Hire #2) — walk through the current router (`forge/0.3/`) and identify the top 3 framework-debt items. Document them in `forge/0.3/framework_debt.md`.
5. **End-of-30 retro** — what blocked, what helped, what to change in the next 30. Co-authored with CTO.

**Trust signal (day 30):** the hire can land a small MCP-server or router change end-to-end and read the SLO dashboard.

## Days 31 – 60 — Ramp + own the framework

**Objective:** own the MCP router + schema registry; unblock the cross-Epic gates.

**Deliverables:**

1. **Cross-Epic gate unblock** — Epic 0.8 (lint), Epic 2.x (architecture), Epic 11.x (sync plane), Epic 9.x (MCP servers). For each: identify the router / SDK gap, file the child issue, drive it `done`. Target: **4 / 4 gates green by day 60**.
2. **Schema registry v1** — declarative source-of-truth in the spirit of Forge AI-119's `api-surface.json`; agents and the router pull schemas, no hard-coded tool lists. Every MCP server's tools registered in the registry by day 60. Backed by a CI lint that fails the build on a missing entry.
3. **Circuit-breaker consistency audit** — every MCP server's circuit-breaker wiring must emit typed `mcp_unavailable` and respect the < 5 s hang budget. Backfill any server that fails the bar. Audit report at `forge/0.3/circuit_breaker_audit.md`.
4. **Per-tenant scope CI lint** — fails CI when a new MCP server PR skips the tenant-scope test. Landed by day 60. Child of Forge AI-48 acceptance #3.
5. **SLO dashboard v1** — per-server p50 / p95, error rate, circuit state, public customer-facing link. Co-owned with DevOps (Hire #4, when active) but the **MCP-side wiring is this hire's deliverable**.
6. **Backlog intake cadence** — every new MCP-server request triaged in 5 business days (per charter §5).
7. **Mid-of-60 retro** — same format as day-30 retro; feed into the day-90 plan.

**Trust signal (day 60):** the hire runs the weekly MCP-server owners' sync and is the default reviewer on router / SDK PRs.

---

## Days 61 – 90 — Lead the platform + shape the next phase

**Objective:** set the platform's direction; ship Wave 3 onboarding; reach framework v1.0.

**Deliverables:**

1. **Wave 3 onboarding** — TestRail, PagerDuty, Linear, or ServiceNow (whichever Epic 11 customer-pull picks first) lands as a drop-in MCP server using the SDK + registry, in < 5 working days from kickoff. Demonstrates the < 10-lines-of-config metric from the charter.
2. **MCP framework v1.0** — router, SDK, schema registry, circuit breakers, scope lint, SLO dashboard — versioned, tagged, with a public changelog. Reuse the DocAgent pattern from [Forge AI-122](/Forge AI/issues/Forge AI-122) for the changelog rendering.
3. **Runbook for new MCP servers** — ≤ 2 pages from "I want to add an MCP server" to "it's in production with SLO + runbook + on-call." Adopted as the team standard.
4. **Hire-loop feedback** — written input to the next `HIRING_PLAN.md` revision: what to keep, what to change for the next platform hire.
5. **End-of-90 retro + first-90 review** with CTO. Decision: extend scope to **MCP Platform Tech Lead** (with a sub-team), or hold and let the framework mature.

**Trust signal (day 90):** the hire proposes the next quarter's MCP roadmap to the CEO; CTO is in the review seat, not the driver seat.
