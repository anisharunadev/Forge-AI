# Forge AI-412 / 0.8.5 — Workspace migration tool with dry-run

**Owner:** CTO (temporary; per Forge AI-103 hiring decision)
**Status:** done
**Acceptance:** [Forge AI-412](/Forge AI/issues/Forge AI-412)
**Blocked by:** [Forge AI-410](/Forge AI/issues/Forge AI-410) — version train (shipped 2026-06-20)

## What this ships

A deterministic, dependency-free migration runner that bumps the
Knowledge Layer schema in a safe, previewable, and idempotent way.
The runner is the smallest credible shape for the spec:

- A new field `appliedMigrations: List[str]` on
  `agents.workspace.version.Manifest` is the source of truth. The
  runner reads it, computes the gap to the user's `--to` target, and
  applies the missing migrations in order.
- Each migration is a small Python package at
  `forge/0.8/migrations/v<N>_<slug>/__init__.py` that exposes
  `version_id`, `description`, `preview(root, manifest)`, and
  `apply(root, manifest)`. Each module owns its own plan and writes;
  the runner is dumb glue.
- `workspace:migrate --to v2_onboarding --dry-run` prints the
  combined plan (adds / deletes / renames / manifest changes /
  tenant override notices). NO files are written.
- `workspace:migrate --to v2_onboarding --apply` calls every
  migration's `apply()` in order, then appends the new version id
  to the manifest's `appliedMigrations`. Re-running `--dry-run` after
  `--apply` reports zero pending migrations (idempotent).
- Tenant overrides under `tenants/<slug>/workspace/...` are NEVER
  written or removed by a migration. The runner inspects those paths
  read-only to surface a shadow NOTICE during preview when a
  migrated seed file would be hidden by a tenant override.

## Design choices surfaced for review

1. **The `appliedMigrations` field is a list, not a counter.** A
   counter (e.g. `migrationVersion: 2`) is the minimum-viable record
   but loses provenance — we cannot tell which migrations ran, only
   that "some sequence up to v2 finished." A list makes the manifest
   grep-able, replayable, and self-documenting. The cost is a few
   extra bytes; the benefit is a manifest a reviewer can read in
   five seconds.

2. **The runner is dumb; the migrations are smart.** Each migration
   module owns its own `preview()` and `apply()`. The runner is
   responsible for ordering, manifest bookkeeping, and protected-path
   enforcement. The split lets a migration author reason about one
   delta in isolation; the runner never has to know the shape of a
   given migration's plan beyond "a list of steps with `op`,
   `relpath`, and (for rename) `old_relpath`."

3. **The runner, not the migration, appends to `appliedMigrations`.**
   This keeps migrations deterministic — `apply()` does the seed
   write, returns, and the runner updates the manifest. A bad
   migration cannot half-write the manifest and half-write the
   seed; the manifest update is a single atomic step after every
   migration in the chain succeeds.

4. **`--apply` on an already-applied target is a no-op, not an
   error.** The spec says "the same end state as a re-`--dry-run`
   after apply"; a no-op `--apply` exits 0 with "0 migration(s)
   pending" rather than failing. An unknown target is still a hard
   error (exit 1). This matches the rollback / version train
   semantics: re-running a known idempotent op is safe; asking for
   something the system has never heard of is not.

5. **Backward-compatible manifest extension.** Adding
   `appliedMigrations` to the manifest is a non-breaking change: a
   manifest written by an older build (no field) loads with
   `applied_migrations=[]`. The schemaVersion stays at 1; the
   migration runner is a layer on top of the manifest, not a
   replacement.

## CLI

```text
python -m agents.workspace.migrate \
    --root workspace/ \
    --migrations-dir forge/0.8/migrations/ \
    --to v2_onboarding \
    --dry-run

python -m agents.workspace.migrate \
    --root workspace/ \
    --migrations-dir forge/0.8/migrations/ \
    --to v2_onboarding \
    --apply

python -m agents.workspace.migrate \
    --root workspace/ \
    --migrations-dir forge/0.8/migrations/ \
    --list
```

Exit codes: `0` success (dry-run or apply), `1` logic error (unknown
target, unknown applied migration on disk, etc.), `2` usage error.

## Critical invariant — tenant overrides survive migrate

`workspace:migrate` NEVER writes to `tenants/<slug>/workspace/...`
or `engagements/<slug>/...`. The check is enforced at three layers:

1. The migration's `apply()` must not write outside `root` — the
   migration author is the first line of defense.
2. The runner validates every plan step in `preview()` via
   `_is_protected_relpath` (the same predicate the version train
   uses for rollback). Any step whose `relpath` starts with
   `tenants/`, `engagements/`, `../tenants/`, or `../engagements/`
   rejects the plan before any I/O.
3. The runner inspects `tenants/<slug>/workspace/<relpath>` for
   every `add` step in the plan; a hit emits a `NOTICE` in the
   preview (the apply is still safe because the seed write does not
   cross the tenant root).

The smoke test (AC3) creates TWO tenant override paths:

- `tenants/acme/workspace/customer/standards.md` — unrelated to
  the new seed file. Byte-identical before and after migrate.
- `tenants/acme/workspace/memory/onboarding.md` — SHADOWS the new
  seed file. The preview emits a NOTICE naming the shadow path;
  the apply is still safe (the seed write is below the tenant
  root); the tenant's version is preserved byte-identical.

## v2_onboarding migration (the demo)

The first migration ships a new Knowledge Layer file at
`memory/onboarding.md`. It is a 10-minute cold-start script for
sub-agents — the document they read BEFORE their stage-specific
memory file. The content:

- A seven-step first-ten-minutes checklist (confirm run + tenant,
  read glossary, read stage memory, read customer conventions, read
  project product requirements document, state plan, honour
  budget/allow-list/audit log).
- A "what you do not do" section (no direct workspace edits, no
  off-allow-list services, no skipped audit log).
- A Related section pointing to the other Knowledge Layer files.

The file passes the Forge AI-408 production-bar lint gate (verified in
isolation against the real seed's glossary). It is a content-only
add: one new file, no renames, no deletes, no manifest schema
change (the manifest is still v1, just with the new
`appliedMigrations` field populated).

Future migrations can rename, delete, or add more files — the
plan data model supports all four ops (`add`, `delete`, `rename`,
`modify`) and the runner validates each.

## Files changed / added

- `agents/workspace/version.py` (modified) — `Manifest` now carries
  `applied_migrations: List[str]` (default `[]`); `load_manifest`
  reads the field; `init` writes it. The change is backward
  compatible: a legacy manifest loads with an empty list.
- `agents/workspace/migrate.py` (new) — ~340 lines, pure stdlib.
  Discovery, loading, planning, applying, rendering, CLI.
- `agents/workspace/smoke_test_migrate.py` (new) — ~290 lines,
  mirrors the Forge AI-408 / Forge AI-410 smoke pattern.
- `agents/workspace/__init__.py` (modified) — extends the package
  docstring to document the migrate runner.
- `forge/0.8/migrations/__init__.py` (new) — package marker; the
  runner imports migration packages from this root.
- `forge/0.8/migrations/v2_onboarding/__init__.py` (new) — the
  first migration; self-contained; ships `memory/onboarding.md`.
- `scripts/smoke.sh` (modified) — adds a `workspace migrate` gate
  after the existing `workspace version` gate. Non-skippable by
  design.

## What this does NOT do (deferred)

- **Manifest schemaVersion bump.** A future v2 of the manifest
  structure (e.g. adding a per-migration timestamp) will need a
  v3 migration. The runner is built for that path, but the v2
  manifest structure is unchanged from v1 — `appliedMigrations` is
  additive.
- **Multi-migration `apply --to vN` skipping intermediates.** The
  runner requires a contiguous applied chain; out-of-order
  targets are rejected. A `--force` flag for skipping
  intermediates is a v0.5 conversation, not a v0.1 blocker.
- **Migration rollback.** The version train handles seed rollback
  (Forge AI-410); the migration runner does not. A migration that
  proves buggy in production is reverted by tagging the
  pre-migration state as vN+1 (per the Forge AI-410 rollback recipe),
  then writing a vN+1 migration that reverts the change. The
  current migrations are content-only adds, so this is a no-op
  exercise until a future migration does a delete or rename.
- **A migration index file.** Migrations are discovered by
  directory listing; there is no central registry. Adding a
  registry is a v0.5 refactor once the migration count grows past
  the "fits in one screen" threshold.

## Risk + rollback

Risk: low. The runner's worst-case failure mode is a partial
apply chain (the first migration wrote, the second crashed) — the
manifest records the partial chain via `appliedMigrations`, and
the operator can either continue the chain by re-running `--apply`
or roll back the seed to the pre-migration tag and re-run. The
runner is read-only in dry-run mode and the smoke gate is
non-skippable, so a regression is caught at the smoke layer before
it reaches a customer.

The Forge AI-410 invariant (tenant overrides survive rollback) is
inherited: the migration runner's `_is_protected_relpath` is the
same predicate the version train uses. AC3 of the smoke test
exercises BOTH paths (unrelated tenant override + shadow tenant
override) and asserts byte-identical content.

To revert this change in an emergency: delete
`agents/workspace/migrate.py`,
`agents/workspace/smoke_test_migrate.py`,
`forge/0.8/migrations/v2_onboarding/`, and
`forge/0.8/migrations/__init__.py`; revert the `version.py`
appliedMigrations edits, the `__init__.py` docstring update, and
the `scripts/smoke.sh` gate. No data on customer systems is
touched — the runner is opt-in and the smoke gate copies the seed
into a tempdir.
