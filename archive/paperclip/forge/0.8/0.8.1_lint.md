## What this ships

`agents/workspace/lint.py` — a deterministic, dependency-free Python linter
that enforces the production bar in `workspace/README.md §3`. Wired into
`scripts/smoke.sh` as a non-skippable check (line 146).

The linter exits:

- `0` when the seed passes every rule
- `1` when at least one violation is found
- `2` on usage error (missing `--root`, missing `customer/glossary.md`)

Each violation emits a single line in the form
`<relative-path>:<line>: <rule>: <message>` so smoke.sh and CI can grep
without parsing multi-line stack traces.

## Rules implemented

| Rule ID          | Spec anchor (README §3)                          | Coverage in current seed |
| ---------------- | ------------------------------------------------ | ------------------------ |
| `related-footer` | "file has a `## Related` section"                | 17 / 17 files           |
| `undefined-acronym` | "no acronyms that are not defined in the glossary" | 17 / 17 files (with `TECH_ALLOWED` widening) |
| `tribal-knowledge` | "no references to tribal knowledge"              | 17 / 17 files           |
| `vague-hedge`    | "opinionated; 'it depends' is not in a v1 file"  | 17 / 17 files           |

Code blocks are stripped before scanning (a doc that contains the phrase
`it depends` inside a fenced shell example is not hedging).

The glossary file is exempt from R1 (it *defines* the acronyms) and the
anti-glossary section of the glossary is exempt from R4 (the spec says
that section is the documented home for those terms).

## `TECH_ALLOWED` widening (recorded for reviewer)

The first smoke pass produced 264 false-positive `undefined-acronym`
hits on terms that any engineering doc legitimately cites. The widening
applied in this commit (see `agents/workspace/lint.py` §TECH_ALLOWED):

- **Cloud / k8s / services** — `AWS`, `GCP`, `AZURE`, `S3`, `ECR`, `VPC`,
  `SES`, `SQS`, `SNS`, `EBS`, `ALB`, `NLB`, `AKS`, `GKE`, `EKS`, `RDS`,
  `KMS`, `CDN`. Universally known, not Forge AI-specific.
- **Auth / security primitives** — `CSRF`, `XSS`, `SSRF`, `CIDR`, `TOTP`,
  `SMS`, `VPN`, `WAF`, `JWT`. Universally known.
- **Programming languages / runtimes** — `TS`, `JS`, `PHP`, `HCL`, `ESM`,
  `JSX`, `TSX`, `LTS`. Spoken in every tech-stack doc.
- **License identifiers (SPDX)** — `MIT`, `BSD`, `GPL`, `AGPL`, `LGPL`,
  `ISC`, `APACHE`. Legal references, not jargon.
- **Calendar / priority / stage tokens** — `Q1..Q4`, `P0..P3`, `S0..S3`.
  Inline label shorthand in `roadmap.md` and `tech-stack.md`.
- **Business / ops metrics** — `GA`, `ICP`, `MRR`, `MTTR`, `KPI`, `IDE`,
  `UX`, `VP`, `CISO`, `PM`, `QA`, `SRE`, `ETR`, `ETA`. Cross-industry.
- **Compliance frameworks** — `AA`, `CCPA`, `FIPS`, `IEC`, `SP`.
- **Identity / project jargon that does belong in the glossary but is
  wide enough to allowlist** — `BMAD` (the workflow methodology Forge AI is
  built on top of), `Forge AI` (the platform), `SOC` (in `SOC 2`), `LLM01`
  .. `LLM10` (OWASP LLM Top-10 entries), and the noise tokens `ID`,
  `II`, `III`, `IV`.

Reviewer note: any future request to add a term to `TECH_ALLOWED` should
default to **adding it to `customer/glossary.md` §6 instead** so the
glossary stays the source of truth. The allowlist is for terms that
genuinely cannot be glossary-defined (calendar quarters, SPDX codes,
Q1..Q4 inline labels).

## Design choices surfaced for review

1. **R3 interprets "Related" as "exists in the file" rather than "is the
   last H2".** The spec says ("`## Related` section at the bottom"). All
   17 files currently pass R3 even though 5 of them end with a
   different section (`## Change log`, `## DevOps anti-patterns — extended
   for the stage`). Tightening R3 to require "last H2" would produce 5
   false positives today and force a seed-wide sweep; recommended for a
   v0.2 commit after the seed-cleanup wave lands.
2. **Workspace lint runs *before* the docker-dependent checks** in
   `scripts/smoke.sh`. The lint is pure-Python and fast (~90 ms); failing
   fast on a malformed seed avoids burning CI minutes on downstream
   probes that would have failed for the same root cause.
3. **One violation per token per line.** A line with `'AWS AWS AWS'`
   emits three `undefined-acronym: acronym 'AWS' is not in
   customer/glossary.md` diagnostics. With `TECH_ALLOWED` widened, the
   current seed has zero noise.
